# Gate 5 evidence ŌĆö p2p connection policy

## L1 ŌĆö Flow E1 (member-as-a-group accept/reject)

**Result:** Soft. `fix_path = TraitImpl`. `fix_effort = Small`.

The `SessionConfig<T>::remote: VerifyingKey` field (confirmed in
`p2panda-sync/src/lib.rs`) is available at session-establish time. A
`PolicyManager<R, F>` wrapper uses a `MemberKeyResolver` (`R`) plus a
caller-supplied reverse-lookup closure (`F: Fn(VerifyingKey) -> Option<MemberId>`)
to authorise or reject the session before delegation.

**Tests:** `spike-p2panda/tests/l1_p2panda_sync.rs`

- `e1_connection_accepted_for_authorised_member` ŌĆö alice in ACL ŌåÆ `policy_check` returns `true`.
- `e1_connection_rejected_member_not_in_acl` ŌĆö bob not in ACL ŌåÆ `PolicyError::Unauthorised`.
- `e1_connection_rejected_unknown_peer` ŌĆö charlie not in reverse index ŌåÆ `PolicyError::UnknownPeer`.
- `e1_connection_rejected_member_revoked_from_trie_via_recheck` ŌĆö session flagged on recheck.

All four tests pass.

### Manager<T> wrapper ŌĆö deferred coverage

A full `impl Manager<T> for PolicyManager` was considered but time-boxed. The
`Manager<T>` trait in `p2panda-sync/src/traits.rs` uses return-position
`impl Trait` (`impl Future`) on all three methods. Wrapping requires either:

1. A concrete adapter struct per `Protocol` associated type (one per concrete
   manager variant), or
2. Boxing all futures (`Pin<Box<dyn Future>>`), which loses zero-cost abstraction.

Neither is disproportionate for a spike, but neither is necessary to confirm the
gate-5 hypothesis. The `policy_check` function is the canonical intercept point;
the wiring to `session()` is a Phase 3 integration task of known small size.
`fix_effort = Small` reflects this.

## L1 ŌĆö Flow E2 (org-as-pseudo-group accept)

**Result:** Soft. `fix_path = TraitImpl`. `fix_effort = Small`.

When the doc ACL is `DocAcl::org_wide()`, `policy_check` delegates to
`resolver.is_member(&member_id)`. This directly leverages the `org_member_ids`
enumeration that was already added to `MemberKeyResolver` for Flow E2/F2
fan-out (per `spike-common/src/resolver.rs`).

**Tests:** `spike-p2panda/tests/l1_p2panda_sync.rs`

- `e2_connection_accepted_via_org_principal` ŌĆö alice and bob (both in trie) accepted; charlie rejected.
- `e2_connection_rejected_member_removed_from_org` ŌĆö bob in reverse index but not in trie ŌåÆ `PolicyError::Unauthorised`.

Both tests pass.

## L2 ŌĆö Flow F1 (termination on member revocation)

**Result:** Soft (pull model only). `fix_path = TraitImpl`. `fix_effort = Small`.

`PolicyManager::recheck_open_sessions()` walks all open session records, calls
`policy_check` for each, and sets `should_close = true` on any session whose peer
is no longer authorised. The caller invokes this after each trie-change event.

**Tests:** `spike-p2panda/tests/l2_g5.rs`

- `f1_terminate_session_on_member_revocation` ŌĆö alice revoked; her session flagged; bob's unchanged.
- `f1_recheck_idempotent_on_already_flagged_sessions` ŌĆö second recheck returns 0 (no double-counting).

Both tests pass.

**Post-open window:** Because the trie is pull-only, a small window exists between
the moment a peer's session is accepted and the next `recheck_open_sessions` call.
This is acceptable for the spike but must be addressed in Phase 3 (see ┬¦Discovered gaps).

**Async close deferred:** Sending `ToSync::Close` to a real p2panda-sync session
requires async + access to the inner `Manager<T>`'s `session_handle(id)` sink.
This is a Phase 3 wiring task; the spike proves the detection logic.

## L2 ŌĆö Flow F2 (org-principal termination on rotation)

**Result:** Soft (same pull model). `fix_path = TraitImpl`. `fix_effort = Small`.

The org-wide ACL path in `recheck_open_sessions` re-evaluates `resolver.is_member`
for each open session. After a member is removed from the org, their session is
flagged; other org members are unaffected.

**Tests:** `spike-p2panda/tests/l2_g5.rs`

- `f2_terminate_org_session_on_member_removal` ŌĆö alice removed from org; alice's session flagged; bob's unchanged.
- `f2_all_sessions_terminated_when_org_emptied` ŌĆö both sessions flagged when org becomes empty.

Both tests pass.

## Discovered gaps in spike-common

### Gap 1: Reverse lookup (VerifyingKey ŌåÆ MemberId)

`MemberKeyResolver` maps `MemberId ŌåÆ P2pMemberKey` (forward direction only).
`SessionConfig<T>::remote` is a raw `VerifyingKey` (the p2panda primitive). The
session-establish flow therefore requires a reverse mapping from `VerifyingKey` to
`MemberId` that is not present in the resolver trait.

**Spike workaround (Option C):** `PolicyManager<R, F>` accepts a
`peer_to_member: F` closure (`F: Fn(VerifyingKey) -> Option<MemberId>`). In tests
the closure is a hand-coded map over known seeds. In production it would be a
secondary structure populated from the trie (e.g. a `HashMap<VerifyingKey, MemberId>`
maintained alongside the primary trie).

**Phase 3 recommendation:** Consider adding a
`find_member_by_key(key: &P2pMemberKey) -> Option<MemberId>` method to
`MemberKeyResolver`, or formalising the reverse-index pattern as a companion struct
(e.g. `KeyIndex`) constructed from the resolver's output. The closure approach works
but leaks a structural concern into every `PolicyManager` instantiation site.

### Gap 2: Trie push notification

`StubTrie` is pull-only. There is no mechanism for the trie to notify the policy
manager when a member is added, removed, or has their key rotated.

**Spike workaround:** `recheck_open_sessions()` is a manual pull: the caller invokes
it after each known trie-change event. This is sufficient for the spike but creates
a post-open window of unauthorised connection.

**Phase 3 recommendation:** Either (a) make the policy manager hold a live
`Arc<RwLock<Trie>>` reference and re-evaluate on each `policy_check` call (always
pulls the latest snapshot), or (b) subscribe the policy manager to a trie-change
notification channel (push model). Option (a) is simpler and is consistent with
the spike's resolver-ownership model.

### Gap 3: Async session close wiring

`PolicyManager` sets `should_close = true` on flagged sessions but cannot actually
close them: sending `ToSync::Close` requires `async` + access to the inner
`Manager<T>`'s `session_handle(id)` sink. This requires the full `Manager<T>`
wrapper to be implemented in Phase 3.

**Phase 3 recommendation:** `PolicyManager` should wrap a concrete
`TopicSyncManager<T, S, L, E>` and, when `recheck_open_sessions` flags a session,
call `session_handle(id).await?.send(ToSync::Close).await?` on each flagged handle.

## Escape hatches

- **Pre-open rejection at the iroh QUIC layer (`Endpoint::builder`):** No
  connection-authorization callback exists (`p2panda-net` upstream issue #925,
  unimplemented in 0.5.2). This would be `Hard` severity and would require a fork
  of `p2panda-net`. Not pursued ŌĆö `Manager<T>` wrapping is the correct salvage path
  within the spike scope.
- **Post-open window:** For high-security deployments, the policy manager could
  check on every message received (not just at session-open), but this is not
  supported by the current `Manager<T>` trait shape without further trait extension.

## Salvage path (fork-locally policy applies)

Per project policy, `fix_path: UpstreamPR` is forbidden. Salvage options in
severity order:

1. **`TraitImpl` (chosen):** Wrap `Manager<T>` with `PolicyManager<M, F>` locally.
   ~150 lines of production code (spike is ~120 lines sans comments). No upstream
   changes needed. This is the gate-5 salvage.

2. **`Fork` (if pre-open QUIC rejection is required):** Fork `p2panda-net` locally
   and add a `ConnectionValidator` callback to `Endpoint::builder`. Estimated effort:
   `Medium` (understand iroh connection lifecycle, add callback hook, keep in sync
   with upstream). Only required if the post-open window is unacceptable for the
   ODS threat model.

Current evidence supports `TraitImpl` as sufficient. The post-open window is
mitigated by fast `recheck_open_sessions` polling (sub-second trie-change latency
is expected in the ODS operational profile).
