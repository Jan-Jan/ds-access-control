## L1 ‚Äî `p2panda-auth` layer (trait override)

**Result:** None (no gap at this layer). `BlockingGroups<Inner>` compiles and intercepts.

### Probe

- `p2panda_auth::traits::Groups` is declared `pub trait Groups<ID, OP, M, C>` with no sealed supertrait or private mechanism. An external `BlockingGroups<Inner>` type that implements `Groups` compiles without special access.
- Every mutation method (`create`, `add`, `remove`, `promote`, `demote`, `receive_from_remote`) is overridable.
- The error type is an associated `type Error: Debug` ‚Äî freely choosable by the implementor.
- The read-only `GroupMembership` trait is separate and NOT affected by the intercept.

### Outcome

`BlockingGroups<Inner>` is implemented in `s2_membership_intercept.rs`. All 5 L1 tests pass:
- `test1` ‚Äî trait is externally implementable (compile-time proof).
- `test2` ‚Äî `add` returns `Err(MutationBlocked)`.
- `test3` ‚Äî `remove`, `promote`, `demote`, `receive_from_remote` all return `Err(MutationBlocked)`.
- `test4` ‚Äî `create` returns `Err(MutationBlocked)`.
- `test5` ‚Äî existing CRDT state remains queryable after blocked mutations (read path unaffected).

**Orphan rule note:** `BlockingGroups` is defined in the local crate, so implementing `Groups` for it is permitted. No newtype wrapping of `Groups` itself is needed.

**Inventory hypothesis status:** Confirmed `Soft`/`None` at auth layer ‚Äî the trait is freely implementable.

## L1 ‚Äî `p2panda-spaces` layer (store wrapper)

**Result:** Hard. `AuthStore<C>` cannot be implemented externally.

### Probe

The design hypothesis was that `TrieGatedAuthStore<Inner>` wrapping `Inner: AuthStore<C>` and refusing `set_auth` calls would be the intercept seam at the spaces layer. Investigation showed this is blocked:

1. `AuthStore<C>` is in `pub mod traits` of `p2panda_spaces` ‚Äî the trait name is reachable.
2. The trait's `set_auth` signature is:
   ```text
   fn set_auth(&self, y: &AuthGroupState<C>) -> impl Future<Output = Result<(), Self::Error>>;
   ```
3. `AuthGroupState<C>` is a type alias in `mod types`, which is a **private module** (`mod types;` in `lib.rs`).
4. `AuthGroupState<C>` expands to `GroupCrdtState<ActorId, OperationId, AuthMessage<C>, C>` where `AuthMessage<C>` is in `mod auth` ‚Äî also a **private module** (`mod auth;` in `lib.rs`).
5. Neither type can be named from outside the crate.

**Compile-time confirmation:**
```text
use p2panda_spaces::types::AuthGroupState;
// ‚Üí error[E0603]: module `types` is private
```

An external impl of `AuthStore<C>` is impossible because the parameter type `AuthGroupState<C>` cannot be named.

### Forge trait secondary assessment

`Forge::forge` takes `SpacesArgs<ID, C>` which IS publicly re-exported (`pub use message::SpacesArgs`). A `BlockingForge` wrapper is syntactically implementable. However, `Forge::forge` is called to produce messages; the `set_auth` call that writes ACL state happens inside `process_local_control` before `forge` is invoked. Blocking `forge` would prevent message creation but would not prevent `set_auth` from being called and potentially writing state if it executes before the forge step. This is not a reliable intercept seam.

**Inventory hypothesis status:** The `Soft`/`TraitImpl` prediction was **wrong** for the spaces layer. The actual severity is **Hard** ‚Äî a private type in the method signature prevents external `AuthStore` impls entirely.

## L2 ‚Äî integrated intercept flow

**Result:** Soft (best-effort). Full end-to-end intercept demonstrated at the `p2panda-auth` CRDT layer; spaces-layer intercept is architecturally blocked (Hard gap at store level).

### What was demonstrated

Three tests in `tests/l2_g2.rs`:

1. **`l2_trie_is_sole_write_authority`** ‚Äî Full flow:
   - Trie-authority path (`GroupCrdt::process`) adds ALICE ‚Äî succeeds.
   - Application code attempts `BlockingGroups::add(MALLORY)` ‚Äî blocked.
   - Final state: ALICE present, MALLORY absent. Invariant confirmed.

2. **`l2_error_type_distinguishable`** ‚Äî `InterceptError::MutationBlocked` is a distinct variant from `InterceptError::Inner(_)`, ensuring callers can distinguish policy-blocks from implementation errors.

3. **`l2_repeated_bypass_attempts_all_blocked`** ‚Äî Five sequential bypass attempts all return `Err(MutationBlocked)`; state is unchanged (only 1 trie-authorised member).

### What was NOT demonstrated (deferred)

The `p2panda-spaces` runtime intercept (via `Group::add` / `Space::add` ‚Üí `set_auth`) was not demonstrated because:
- `AuthStore<C>` cannot be implemented externally (see L1 finding above).
- Without a conforming `AuthStore<C>`, `Manager<‚Ä¶>` / `Group<‚Ä¶>` / `Space<‚Ä¶>` cannot be instantiated.
- This is an architectural blocker, not a time-box issue.

## Salvage path

Per project policy, `UpstreamPR` is not available. The fix path for the Hard gap is `Fork`.

**Hard gap ‚Äî spaces store layer:**
Fork `p2panda-spaces` locally. Add one line to `lib.rs`:
```text
pub use types::AuthGroupState;
```
This makes `AuthGroupState<C>` nameable from outside the crate, unlocking external `AuthStore<C>` impls and the `TrieGatedAuthStore` wrapper. The `AuthMessage<C>` type in the `set_auth` parameter position does not need to be named if the impl uses `async fn` syntax (the compiler infers the argument type from the trait).

**Alternative (smaller footprint):** Instead of the `TrieGatedAuthStore` approach, use the `Forge` trait to intercept message creation (blocking forging when the message corresponds to an auth mutation). This is a weaker guarantee (state may be updated internally before forge is called) but does not require naming `AuthGroupState`.

**Fix effort assessment:** `Small` ‚Äî one re-export line + `TrieGatedAuthStore` (~15 LOC) in a local fork. The fork itself adds maintenance cost; see `project_p2panda_fork_policy` memory for the fork governance model.

**Aggregate severity for gate 2:**
- `p2panda-auth` layer: `None` (fully interceptable via `BlockingGroups`).
- `p2panda-spaces` layer: `Hard` (private type blocks external `AuthStore` impl).
- Aggregate: `Hard` (worst layer), `fix_path = Fork`, `fix_effort = Small`.

The Hard is narrower than gate 1's Hard: the root cause is a single missing public re-export (one line), not a deep architectural coupling. The fork effort is minimal.
