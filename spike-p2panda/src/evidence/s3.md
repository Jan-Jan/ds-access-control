# Gate 3 evidence — (D)CGKA recompute on trie change

## L1 Flow B (compute via resolver)

**Result:** Soft.

`Dcgka<ID, OP, PKI, DGM, KMG>` is fully generic over `PKI`. The `PKI` type
must satisfy BOTH `IdentityRegistry<ID, PKI::State>` AND
`PreKeyRegistry<ID, LongTermKeyBundle>` — and crucially, `DcgkaState::pki` is
of type `PKI::State`, which is shared between the two traits. The library's
own `KeyRegistry<ID>` uses `KeyRegistryState<ID>` as that shared state and
satisfies both traits cleanly.

`KeyRegistry::add_longterm_bundle` is the injection point: the adapter calls
`resolver.p2p_member_key(id)` to get the current key, builds a `LongTermKeyBundle`
(using `KeyManager::init_and_generate_prekey`), and registers it. The identity key
stored in `KeyRegistryState::identities` is then sourced from the bundle's
`identity_key()` field — completing the resolver → DCGKA bridge.

`Dcgka::create` + `Dcgka::process` succeed with `KeyRegistry<DcgkaMemberId>` as
`PKI`, confirmed by `flow_b_dcgka_create_and_process_shared_secret` (5/5 L1 tests
pass). The two-member group produces an identical `GroupSecret` for Alice and Bob.

**Friction:** `IdentityHandle` (and `OperationId`) are not implemented for
`spike_common::MemberId` because both are foreign types (orphan rule). Gate 3
introduces `DcgkaMemberId` — a newtype over `MemberId` defined in this crate —
as the local `IdentityHandle` type. This is a one-time ~10-line addition.

**Friction:** `DcgkaError<ID, ...>` uses `thiserror` with `#[error("... {0}")]`
on the `ID` field, requiring `ID: Display`. `DcgkaMemberId` implements `Display`
(first 4 bytes in hex). This is a minor addition but worth noting: any production
`ID` type used with `Dcgka` must implement `Display`.

**Friction:** `KeyManager::init_and_generate_prekey` is behind `#[cfg(any(test,
feature = "test_utils"))]` in p2panda-encryption. The setup helpers
(`build_member_states`, `init_g3_dcgka_state`) therefore live in the test files
rather than the library, activated via `dev-dependencies` adding
`features = ["data_scheme", "test_utils"]` to `p2panda-encryption`. In production,
members publish their own `LongTermKeyBundle` and the adapter registers it via
`KeyRegistry::add_longterm_bundle`; the `test_utils` path is only needed for
local bundle generation in spike tests.

## L1 Flow C (recompute on rotation)

**Result:** Soft.

Two levels of evidence:

**Level 1 — resolver layer:** `IdentityRegistry::identity_key` on
`ResolverPki<StubTrie>` returns a different x25519 key after the trie is updated
with a new `p2p_member_key`. Confirmed by `flow_c_identity_key_reflects_trie_rotation`.
The resolver is live — no internal cache.

**Level 2 — DCGKA layer:** `trigger_recompute` injects a fresh
`KeyRegistryState<DcgkaMemberId>` (simulating a resolver-triggered PKI rebuild
after rotation), calls `Dcgka::update`, and produces a new `GroupSecret`. Bob
processes the update and obtains the same new secret. The epoch-0 and epoch-1
secrets are distinct. Confirmed by `flow_c_dcgka_recompute_yields_new_group_secret`.

The `trigger_recompute` function (~15 lines in `s3_cgka_rotation.rs`) is the
production hook: on trie observer notification, the adapter: (1) resolves updated
member keys via the resolver, (2) rebuilds the `KeyRegistry` with new bundles,
(3) calls `trigger_recompute`, (4) distributes the resulting direct messages
to group members.

**Note:** The spike simulates key rotation by building an entirely fresh
`KeyRegistry` (new `KeyManager` per member). In production, rotation involves:
the member generating and publishing a new `LongTermKeyBundle`, the adapter
receiving it via the trie/network, and calling `KeyRegistry::add_longterm_bundle`
to update the registry before `trigger_recompute`.

## L2 integrated flow

**Result:** Soft.

`l2_g3_combined_flow_b_and_c` runs the full pipeline:
1. 2-member DCGKA group created with registry-populated key material (Flow B).
2. Alice and Bob share the epoch-0 `GroupSecret`.
3. `trigger_recompute` with fresh PKI → epoch-1 `GroupSecret` produced.
4. Epoch-0 ≠ epoch-1 secrets (forward-secrecy signal).
5. Alice and Bob share the epoch-1 secret after Bob processes the update.
6. `ResolverPki::identity_key` reflects the pre- and post-rotation keys.

All 6 L2 assertions pass.

## Escape hatches

- **`DcgkaMemberId` newtype:** Required to implement `IdentityHandle` for
  `MemberId` without orphan rule violation. ~10 lines; straightforward.
- **`DcgkaMemberId: Display`:** Required by `DcgkaError`'s `thiserror` macros.
  ~5 lines; straightforward.
- **`LocalDgm<ID, OP>`:** Mirrors upstream `TestDgm` but is defined locally so
  the production library does not require the `test_utils` feature of
  `p2panda-encryption`. ~60 lines; straightforward.
- **`test_utils` feature in dev-dependencies:** `KeyManager::init_and_generate_prekey`
  gated on `test_utils`. Dev-dep feature activation enables it for tests without
  leaking into production builds.
- **Shared PKI state type:** `DcgkaState::pki` is the SAME type used as both
  `IdentityRegistry::Y` and `PreKeyRegistry::State`. This means the custom
  `ResolverPki<R>` adapter (from gate 1) CANNOT be used as `PKI` directly —
  it would require `Serialize + Deserialize` on the resolver `R`, and its
  `PreKeyRegistry::State` would need to carry pre-key bundle storage. The cleaner
  path (used here) is `KeyRegistry<DcgkaMemberId>` as `PKI`, populated from
  resolver data. `ResolverPki` remains the gate-1 identity-key seam; the DCGKA
  layer uses `KeyRegistry` directly. This is architecturally correct: the resolver
  is called at group-setup and rotation time to populate the registry, not on every
  DCGKA operation.

## L3 integration finding — `Dcgka::remove` vs `trigger_recompute` for revocation

When demonstrating the revocation scenario end-to-end (`tests/l3_revocation.rs`),
the correct DCGKA operation is `Dcgka::remove`, NOT `trigger_recompute`
(which wraps `Dcgka::update`).

`Dcgka::update` sends direct messages to every member listed in the
`AckedGroupMembership` (DGM), including the member being removed. That
breaks the design's forward-security requirement: the revoked member's
device receives the key material it would need to derive the new group
secret.

`Dcgka::remove` explicitly filters out the removed member from the
direct-message fan-out. The post-removal group secret is unreachable
from the revoked member's local state.

`trigger_recompute` (gate 3, Flow C) remains the correct operation for
the key-rotation flow — all current members should receive the new secret
after rotation. Phase 3 will expose both operations: one for key rotation
via `trigger_recompute`, one for membership removal via `Dcgka::remove`.

Evidence: `spike-p2panda/tests/l3_revocation.rs` (doc comment + test body).

## Salvage path

**No `UpstreamPR` per project policy.**

Severity is `Soft` — no blocking friction. All required generics are public and
externally implementable. Salvage plan for Phase 3:

1. **`DcgkaMemberId` adapter:** ship as-is; ~10 lines; stays in the adapter layer.
2. **`LocalDgm`:** replace with the real p2panda-auth `Groups` CRDT in Phase 3
   (gate 2 showed the CRDT layer is externally implementable).
3. **PKI population from resolver:** the `build_member_states` test helper
   documents the production pattern. Phase 3 implementation: on trie notification
   → resolve new bundle → `KeyRegistry::add_longterm_bundle` → `trigger_recompute`.
4. **`test_utils` gate:** not a Phase 3 concern; production members publish their
   own `LongTermKeyBundle` via the network protocol.
5. **DCGKA O(n) scaling:** noted in inventory (§Addendum). For large groups,
   consider batching `trigger_recompute` calls or coalescing rotations per epoch.
   Not a friction for gate 3 — the API is correct.
