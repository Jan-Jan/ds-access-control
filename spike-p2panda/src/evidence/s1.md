# Gate 1 evidence — stable-ID ACL

## L1 — `p2panda-auth` layer (Task 3)

**Date:** 2026-05-28
**Pinned commit:** 41559b0
**Test file:** `spike-p2panda/tests/l1_p2panda_auth.rs`

### What was tested

**Test 1 — `MemberId` satisfies `IdentityHandle` bound.**
Defined a local newtype `SpikeMemberId([u8; 32])` (same layout as `spike_common::MemberId`) and implemented `IdentityHandle` for it. Passing `SpikeMemberId` instances to a helper bounded by `IdentityHandle` is the compile-time proof. Additionally verified that `spike_common::MemberId` can be converted into `SpikeMemberId` via `From`, confirming the byte representation is identical.

**Test 2 — Construct `GroupCrdtState<SpikeMemberId, …>` and exercise `add`.**
Built a minimal synchronous `SpikeOp` that implements the `Operation<SpikeMemberId, OpId, ()>` trait without any networking, signing, or async machinery. Used `GroupCrdt::process` to drive the CRDT: first a `Create` operation (Alice as initial manager), then an `Add` operation (Alice adds Bob with `Manage` access). Verified membership via both `members()` (transitive) and `root_members()` (direct).

**Test 3 — Stable-ID property (ACL serialisation embeds `SpikeMemberId`, not ed25519 keys).**
Two-part test:
- *Part A (in-memory):* Confirmed that `root_members()` and `members()` return `SpikeMemberId` values carrying exactly the bytes we inserted (`0xa1` for ALICE, `0xb1` for BOB). No `VerifyingKey` type appears anywhere.
- *Part B (CBOR round-trip):* Serialised an `SpikeOp` to CBOR using `ciborium` (the same serialiser the upstream tests use). Confirmed that bytes `0xa1` and `0xb1` appear in the CBOR output and that the op round-trips cleanly.

Sample CBOR output (first 48 bytes of op1):
```text
[a5, 62, 69, 64, 01, 66, 61, 75, 74, 68, 6f, 72,
 98, 20,                          <- 32-element byte array follows
 18, a1, 18, a1, 18, a1, 18, a1,  <- [0xa1; 32] = ALICE's SpikeMemberId
 18, a1, 18, a1, 18, a1, 18, a1,
 ...]
```
`18 a1` is CBOR for the integer 161 (= 0xa1); the 32-element array is the `SpikeMemberId` stored verbatim.

### What the API actually exposed at this revision

The inventory's API descriptions were accurate, with the following clarifications and corrections:

| Inventory claim | Actual at 41559b0 |
|---|---|
| `Groups` trait with `add(group_id, adder, added, Access<C>)` | The `Groups` trait exists in `p2panda_auth::traits`, but the CRDT entry point is `GroupCrdt::process(state, &op)` — not a method on `Groups` directly. `Groups` is the trait for a higher-level, state-carrying API; the raw CRDT uses `GroupCrdt::<…>::process`. |
| `GroupMember<ID>` — `Individual(ID) \| Group(ID)` | Correct. |
| `IdentityHandle` bound — `Copy + Debug + PartialEq + Eq + Ord + StdHash` | Correct; confirmed in `src/traits/mod.rs`. |
| `Access::manage()` constructor | Correct. |
| `processor` feature = default | Confirmed: `default = ["processor"]`; pulls in tokio, p2panda-store, p2panda-stream. The L1 test uses only `serde` feature (no `processor`, no `test_utils`). |
| `test_utils` feature exports helpers | Correct, but they also pull in `p2panda-stream` which depends on tokio. We avoided this by implementing `Operation` locally. |

**Orphan rule note:** `impl IdentityHandle for MemberId` cannot be written in a downstream test crate because both `IdentityHandle` and `MemberId` are defined in external crates. A newtype wrapper `SpikeMemberId` (same layout) is required. This is a minor friction but not a blocker — in Phase 3, `impl IdentityHandle for MemberId` would be written in the `spike-p2panda` library crate where `MemberId` is re-exported or re-defined locally.

### Findings

- **`MemberId` satisfies `IdentityHandle` bound:** YES — confirmed at compile time. `MemberId` derives `Copy + Debug + PartialEq + Eq + PartialOrd + Ord + Hash`, which exactly covers the `IdentityHandle` bound. The orphan rule requires a newtype in the test crate but not in the integration crate.

- **Constructing `GroupCrdtState<SpikeMemberId, …>` and exercising `add`:** PASS. The CRDT is straightforward to instantiate without any async, store, or network machinery. `GroupCrdt::process` takes owned state + `&op` and returns new state. The `StrongRemove` resolver is available without any feature flags beyond `default-features = false, features = ["serde"]`.

- **Stable-ID property (ACL serialisation embeds `SpikeMemberId`, not a raw key):** PASS. At the `p2panda-auth` layer, the ACL is stored exclusively in terms of the generic `ID` parameter. No `VerifyingKey` type is present in the CRDT state at this layer. When `ID = SpikeMemberId([u8; 32])`, the serialised form contains only 32-byte ID arrays.

### Escape hatches used

1. **`ciborium` as dev-dependency** — `serde_json` cannot serialise `HashMap<OpId, …>` (non-string key constraint). `ciborium` (CBOR) is used for the serialisation round-trip in Test 3, matching the upstream test convention.
2. **`serde` as dev-dependency** — needed to derive `Serialize`/`Deserialize` for `SpikeMemberId`, `OpId`, and `SpikeOp` in the test file.
3. **`SpikeMemberId` newtype** — required by the orphan rule (cannot impl a foreign trait for a foreign type). Same layout as `MemberId`; same derives.
4. **No `test_utils` feature** — avoided to keep compile time low and avoid pulling in `p2panda-stream`/tokio in the test binary. The `Operation` and `Resolver` traits are implemented locally.

### Gate-1 hypothesis status (post-L1)

The inventory hypothesis was: *"Soft or None at the `p2panda-auth` layer; Hard at `p2panda-spaces` layer."*

**L1 at `p2panda-auth` confirms: None (no gap at this layer).**

- `GroupMember<ID>` is fully generic over any `ID: IdentityHandle`.
- `MemberId` satisfies `IdentityHandle` (orphan rule aside, trivially satisfied).
- The ACL is stored, serialised, and queried entirely in terms of the opaque ID.
- No ed25519 key material leaks into the CRDT state at this layer.

The Hard gap at `p2panda-spaces` (where `ActorId` is hardwired over `VerifyingKey`) remains the key concern — to be probed in Task 4 (L1 at `p2panda-spaces`).

## L1 — `p2panda-spaces` layer (Task 4)

**Date:** 2026-05-28
**Pinned commit:** 41559b0
**Test file:** `spike-p2panda/tests/l1_p2panda_spaces.rs`

### What was tested

**Probe 1 — Direct substitution (compile-time).**
Confirmed that `Group::add` and `Space::add` are concretely typed to `ActorId` and
cannot accept any stable identity type (e.g. `MemberId`) in their place. Documented
the cannot-compile snippet inline in the test file. Verified that `ActorId` is an
ed25519-specific type (192 bytes in memory — the full `ed25519_dalek::VerifyingKey`
struct with expanded representation), not a raw opaque 32-byte buffer.

**Probe 2 — `ActorId` construction (runtime).**
Enumerated all public constructors of `ActorId` from outside the `p2panda-spaces` crate.
Confirmed which paths exist and which don't. Key finding: `From<VerifyingKey>` is public
(surprise vs. original inventory expectation), but there is no `From<MemberId>`. The
stable-ID-to-key translation must happen externally before the spaces API is called.

**Probe 3 — Stale-key behaviour (runtime).**
Skipped due to scaffolding cost (full async store stack, Forge impl, key machinery).
Compile-time evidence from probes 1+2 is sufficient; Task 5 (L2 integrated) will
provide the runtime confirmation.

### What the API actually exposed at this revision

**`ActorId` definition** (`p2panda-spaces/src/types.rs` at 41559b0):
```text
#[derive(Clone, Copy, Debug, PartialEq, Eq, PartialOrd, Ord, Hash, Serialize, Deserialize)]
pub struct ActorId(pub(crate) VerifyingKey);
```
Inner field is `pub(crate)` — tuple struct literal is inaccessible from outside the crate.

**Public constructors:**
```text
impl From<VerifyingKey> for ActorId { ... }         // public — exists (surprise)
impl TryFrom<[u8; 32]> for ActorId { ... }          // public — exists
impl TryFrom<&[u8]> for ActorId { ... }             // public — exists
pub fn from_bytes(bytes: &[u8; 32]) -> Result<…>   // public — exists
// ActorId::new(vk)                                 // DOES NOT EXIST
// ActorId(vk) tuple literal                        // COMPILE ERROR — field pub(crate)
```

**`Group::add` signature** (`p2panda-spaces/src/group.rs` at 41559b0):
```text
pub async fn add(
    &self,
    member: ActorId,       // concrete — not generic over IdentityHandle
    access: Access<C>,
) -> Result<(Vec<M>, Vec<Event<ID, C>>), GroupError<ID, S, K, F, M, C, RS>>
```

**`Space::add` signature** (`p2panda-spaces/src/space.rs` at 41559b0):
```text
pub async fn add(
    &self,
    member: ActorId,       // concrete — not generic over IdentityHandle
    access: Access<C>,
) -> Result<(Vec<M>, Vec<Event<ID, C>>), SpaceError<ID, S, K, F, M, C, RS>>
```

**`Manager`'s `ID` generic** is the *space* identifier (`ID: SpaceId`), NOT the member
identity. All `AuthGroup`, `EncryptionGroup`, and related type aliases in `types.rs`
concretise over `ActorId` for the member-identity position.

**Cannot-compile evidence:**
The following snippet (not in the test file, which must compile) would fail:
```text
// Hypothetical: using MemberId-equivalent type with spaces Group::add
fn add_stable_member<I: p2panda_auth::traits::IdentityHandle>(
    group: &p2panda_spaces::Group<SomeSpaceId, S, K, F, M, C, RS>,
    member: I,
    access: p2panda_auth::Access<()>,
) {
    // ERROR: expected `ActorId`, found type parameter `I`
    let _ = group.add(member, access); // <-- compile error
}
```
rustc error (conceptual):
```text
error[E0308]: mismatched types
  --> ...
   | expected `ActorId`, found type parameter `I`
   = note: `I` is a generic parameter, not `ActorId`
```

### Findings

- **Probe 1 (direct substitution):** HARD — `Group::add` and `Space::add` take
  `ActorId` concretely. No generic parameter flows through to the member-identity
  position. `MemberId` (or any stable-ID type) cannot be passed to these methods.

- **Probe 2 (ActorId construction):** `From<VerifyingKey>` is public (surprise — the
  original inventory implied the field might be fully private). This means building an
  `ActorId` from a trie-resolved `VerifyingKey` is feasible in one step. However, the
  key-to-member reverse lookup (stale-key detection) remains entirely the caller's
  responsibility. No path `From<MemberId>` exists.

  In-memory size of `ActorId`: **192 bytes** — the full `ed25519_dalek::VerifyingKey`
  struct with compressed + expanded Montgomery form. Not a raw opaque buffer.

- **Probe 3 (stale-key behaviour):** SKIPPED — scaffolding cost too high for the
  allotted time-box. Analysis: because `Group::add(member: ActorId, ...)` pins to a
  concrete `VerifyingKey` at call time, any ACL state built via this path records the
  *key value* provided. Key rotation requires the caller to explicitly remove the old
  `ActorId(K1)` and add `ActorId(K2)`. `p2panda-spaces` has no internal mechanism to
  track or react to key rotation. Task 5 will confirm at runtime.

### Salvage path (Hard)

The inventory hypothesis identified `TraitImpl` as the salvage path. Based on probe
results, this remains correct and is tractable:

1. **Use `p2panda-encryption::IdentityRegistry` directly.** Rather than routing through
   `Space::add` / `Group::add`, implement `p2panda-encryption::IdentityRegistry<MemberId>`
   that holds `MemberId`-indexed key bundles. Populate this registry from
   `MemberKeyResolver` (trie lookups). This gives `Dcgka<MemberId, …>` a clean
   stable-ID-indexed PKI, bypassing the spaces API entirely for the key-lookup concern.

2. **Implement a thin wrapper around the spaces manager.** Expose an
   `add_member(member_id: MemberId)` API on the wrapper that: (a) resolves
   `MemberId → VerifyingKey` via the trie, (b) constructs `ActorId::from(verifying_key)`,
   (c) calls `Group::add(actor_id, access)`. This is straightforward because
   `From<VerifyingKey>` is public (probe 2 finding).

3. **Handle key rotation in the wrapper.** On trie-change events (K1 → K2): wrapper calls
   `Group::remove(ActorId::from(K1))` then `Group::add(ActorId::from(K2), old_access)`.
   This is the "TraitImpl" fix — it's boilerplate but not architecturally blocked.

The salvage is `fix_effort = Medium` (not Small) because the rotation logic, concurrent
update handling, and integration with the async spaces stack require non-trivial
engineering, even though there are no hard blockers.

### Gate-1 hypothesis status (post-L1 at spaces)

The inventory hypothesis was: **Hard at `p2panda-spaces`, salvage = TraitImpl**.

**Confirmed: Hard at `p2panda-spaces` layer.**

- `Group::add` and `Space::add` are concretely typed to `ActorId` — no generic member-ID.
- All type aliases in `types.rs` fix the member-identity position to `ActorId`.
- The `Manager`'s `ID` generic is for space identity, not member identity.
- `From<VerifyingKey> for ActorId` is public (minor positive surprise — makes the wrapper
  path slightly less painful), but does not change the Hard severity.
- No ed25519-free path to `Group::add` exists — a `VerifyingKey` is always required.

**Aggregated gate-1 severity for the p2panda library: Hard (worst layer).**
- `p2panda-auth` layer: None (fully generic, clean).
- `p2panda-spaces` layer: Hard (ActorId hardwired, no generic member-ID at the user-facing API).
- Gap-matrix row (Panda, 1, A, Member) updated to severity Hard, failing_subcrate = p2panda-spaces.

## L2 — Integrated stable-ID ACL flow (Task 5)

**Date:** 2026-05-28
**Pinned commit:** 41559b0
**Implementation:** `spike-p2panda/src/s1_stable_id_acl.rs`
**Tests:** `spike-p2panda/tests/l2_g1.rs`

### What was implemented

Both pieces from the Task 5 plan were implemented:

**1. `ResolverPki<R>` — encryption-layer PKI adapter.**
A struct that wraps any `MemberKeyResolver` and implements
`IdentityRegistry<MemberId, ResolverPki<R>>` from `p2panda-encryption`.
The `Y` (state) type parameter is `ResolverPki<R>` itself — the resolver is threaded
via the state argument of the static-method `identity_key(y: &Y, id: &ID)`. Instance
methods `identity_key_with_resolver` and `org_identity_key` provide the ergonomic
call path without the static-method indirection.

**2. `materialise_actor_id` — spaces-layer call-time converter.**
A free function accepting any `MemberKeyResolver` and a `Principal`, returning a
freshly-materialised `ActorId` by resolving the current `VerifyingKey` and calling
`ActorId::from(PandaVerifyingKey::from(dalek_vk))`. This is the integration seam
between stable-ID application code and the `p2panda-spaces` API.

### Flow A end-to-end demonstration

Seven tests in `tests/l2_g1.rs` exercise the full Flow A path:

- **Test 2 (rotation tracking):** Alice starts with key K1. `materialise_actor_id` returns
  `ActorId(K1)`. Trie is updated to K2 via `stub_rotate_member_key`. The next call to
  `materialise_actor_id` returns `ActorId(K2)`. `assert_ne!(actor_before, actor_after)` PASSES.
  This is the runtime confirmation of Probe 3 from Task 4 (previously skipped).

- **Test 7 (IdentityRegistry rotation):** Same rotation scenario confirmed at the
  `IdentityRegistry::identity_key` trait layer — the static method returns K2 after rotation.

The rotation-tracking property holds because both `materialise_actor_id` and
`ResolverPki::identity_key` call the resolver on every invocation. There is no internal
cache; the trie is the single source of truth.

All 7 tests passed. Workspace tests, `clippy --all-features`, and `clippy --tests` all exit 0.

### Escape hatches used

1. **`IdentityRegistry::identity_key` is a static method (no `&self`).**
   The upstream trait signature at 41559b0 is:
   ```text
   fn identity_key(y: &Y, id: &ID) -> Result<Option<PublicKey>, Self::Error>
   ```
   No `&self` receiver. To give the impl access to the stored resolver, the `Y` (state)
   type parameter is set to `ResolverPki<R>` itself — the caller passes `&pki` as `y`.
   This is unconventional but valid. A production Phase-3 integration should either (a)
   adopt the `Y`-carries-context pattern consistently, or (b) propose an upstream
   `&self`-receiver change to `IdentityRegistry`.

2. **Ed25519 → X25519 byte reinterpretation.**
   `IdentityRegistry::identity_key` returns `p2panda_encryption::crypto::x25519::PublicKey`
   (a 32-byte ECDH key). `P2pMemberKey` wraps `ed25519_dalek::VerifyingKey` (32 bytes
   compressed). `ResolverPki` reinterprets the compressed ed25519 bytes as an x25519
   `PublicKey` via `PublicKey::from_bytes`. This is safe for the PKI-lookup role (the
   identity key is a stable anchor for bundle verification; actual ECDH uses pre-key
   bundles). In a full Phase-3 deployment the two key systems would be properly separated
   (separate ed25519 signing key and x25519 ECDH key per member).

3. **`p2panda_core::identity::VerifyingKey` as bridge type.**
   `ActorId::from` requires `p2panda_core::identity::VerifyingKey`, not
   `ed25519_dalek::VerifyingKey` directly. `p2panda_core` provides
   `impl From<ed25519_dalek::VerifyingKey> for VerifyingKey`, so the conversion is:
   `dalek::VerifyingKey → PandaVerifyingKey → ActorId` — two trivial newtype steps.

4. **No `MemberKeyResolver` impl for `&StubTrie`.**
   `spike-common` only implements `MemberKeyResolver for StubTrie` (owned). Tests that
   need to pass `&trie` use `trie.clone()` or consume the trie. In Phase 3 a blanket
   `impl<T: MemberKeyResolver> MemberKeyResolver for &T` would be added to `spike-common`.

### Resolved severity

The gap-matrix row at Task 4 was **Hard**. The L2 evidence does not change this
assessment:

- **Still Hard.** The `p2panda-spaces` API (`Group::add`, `Space::add`) hardwires
  `ActorId` with no generic member-ID parameter. The `materialise_actor_id` wrapper
  and `ResolverPki` are genuine TraitImpl salvage additions — they are not cosmetic
  workarounds but real glue code that must be written and maintained.

- **Not showstopping.** The salvage is tractable: the two pieces together are ~120 lines
  (lib module + test file = ~310 lines total, of which ~70 are functional code). The
  rotation-tracking logic is simple (call-time resolution). No async scaffolding, no
  store stacks, no signing machinery was needed for the L2 demonstration.

- **The Hard rating reflects friction, not a blocker.** The gap is "annoying" (caller
  must manage `remove + add` on rotation) rather than "showstopping" (there is no path
  at all). `fix_effort` is refined to `Small` from `Medium` based on actual L2 LOC:
  the core salvage code is ~40 functional lines, substantially less than the original
  `Medium` estimate.
