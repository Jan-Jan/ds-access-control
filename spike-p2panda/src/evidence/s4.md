## L1 ŌĆö p2panda-auth (`GroupMember::Group(ID)`)

**Result:** None (no gap at this layer).

`GroupMember::Group(ID)` exists and works exactly as needed to model the
org-as-pseudo-group pattern. The relevant code paths at 41559b0:

**`GroupMember` enum** (`p2panda-auth/src/group/member.rs`):
```text
pub enum GroupMember<ID> {
    Individual(ID),
    Group(ID),     // ŌåÉ the nested-group variant
}
```
Both variants are generic over the same `ID: IdentityHandle`. There is no
separate type for group IDs vs individual IDs ŌĆö a group's own identifier is a
`SpikeMemberId` value just like a member's identifier.

**`GroupAction::Add` accepting `GroupMember::Group`**:
In `p2panda-auth/src/group/crdt/mod.rs` (lines 683ŌĆō689), `apply_action` passes
`GroupMember::Individual(actor)` as `adder` and the raw `member` field (which
can be `GroupMember::Group(id)`) as `added`. `GroupCrdt::process` accepts an
`Add` op with `member: GroupMember::Group(ORG_GID)` without modification.

**Auto-resolution of nested membership** (`crdt/mod.rs` lines 201ŌĆō258,
`members_inner`):

```text
match member {
    GroupMember::Individual(id) => { members.entry(id).or_insert(next_access); }
    GroupMember::Group(id) => self.members_inner(id, members, Some(next_access), depth),
}
```

`GroupCrdtState::members(doc_gid)` auto-resolves nested group membership ŌĆö
it recursively follows `GroupMember::Group(id)` entries and returns a flat
`Vec<(ID, Access<C>)>` of individual members. **No manual walking is needed in
Phase 3's org-pseudo-group adapter for membership queries.**

**One constraint**: groups added with `Access::manage()` are rejected
(`GroupCrdtError::ManagerGroupsNotAllowed`). The org pseudo-group must be added
with `Access::read()` or `Access::write()`. This is expected and acceptable for
the ODS use-case (the org grants read/write access; individual managers hold
`Manage` access via their `Individual` membership entries).

**Test evidence** (all in `tests/l1_p2panda_auth.rs`):
- Test 4: `GroupAction::Add { member: GroupMember::Group(ORG_GID), access: Access::read() }`
  accepted by `GroupCrdt::process`. `root_members(DOC)` returns
  `[GroupMember::Individual(ALICE), GroupMember::Group(ORG_GID)]`.
- Test 5: `members(DOC)` returns `[(ALICE, Manage), (BOB, Read)]` ŌĆö BOB was
  only added to ORG_GID, never to DOC_GID. Auto-resolution confirmed.
- Test 6: `GroupMember::Group(ORG_GID)` with `Access::manage()` is rejected.

**Inventory hypothesis status:** Confirmed `Soft` at auth layer ŌĆö but the
actual finding is stronger than `Soft`. The variant exists, is generic over
`ID`, is accepted by `GroupCrdt::process`, and `members()` auto-resolves it.
Severity revised to **None** (no gap at this layer).

## L1 ŌĆö p2panda-spaces (no nested-group support)

**Result:** Hard (cascades from gate 2 finding).

`Group::add` and `Space::add` in `p2panda-spaces` accept only `ActorId`
(concrete, hardwired). There is no API to add a `GroupMember::Group` reference.

**Exact signature** (`p2panda-spaces/src/group.rs` at 41559b0):
```text
pub async fn add(
    &self,
    member: ActorId,       // concrete ŌĆö NOT generic over GroupMember or IdentityHandle
    access: Access<C>,
) -> Result<(Vec<M>, Vec<Event<ID, C>>), GroupError<ID, S, K, F, M, C, RS>>
```

`ActorId` is defined as `struct ActorId(pub(crate) VerifyingKey)` ŌĆö it wraps
a concrete ed25519 key. There is no `ActorId` representation of a group
reference; `ActorId` is always an individual.

This gap is a direct cascade from gate 2's finding: `AuthStore<C>` uses
`AuthGroupState<C>` from the private `mod types` in p2panda-spaces, making it
impossible to implement `AuthStore` externally without a fork. The nested-group
restriction is a deeper expression of the same coupling: the spaces integration
layer is concretised over individual `ActorId` values throughout.

**No runtime test needed**: the constraint is structural ŌĆö `Group::add` cannot
be called with a group reference from outside the crate.

## L2 ŌĆö Flow A + Flow C end-to-end

**Result:** Soft (demonstrated via `p2panda-auth` CRDT; spaces layer bypassed).

Two tests in `tests/l2_g4.rs`:

### Flow A ŌĆö `flow_a_org_principal_grants_both_members_access`

Setup:
- `ORG_GID` group: `ORG_MANAGER` (Manage) + `ALICE` (Manage) + `BOB` (Manage)
- `DOC_GID` group: `DOC_MANAGER` (Manage) + `ORG_GID` (Group, Read)
- StubTrie: alice and bob registered with ed25519 keys K_alice and K_bob

Assertions:
1. `state.members(DOC_GID)` returns both `ALICE` and `BOB` (auto-resolution).
2. `effective_member_keys(&state, DOC_GID, &trie)` returns `{K_alice, K_bob}`.
3. `ORG_MANAGER` and `DOC_MANAGER` are absent from the key set (not in trie).

Result: PASS.

### Flow C ŌĆö `flow_c_org_principal_tracks_alice_rotation`

Setup: same as Flow A.

Steps:
1. Before rotation: `effective_member_keys` returns `{K1_alice, K_bob}`.
2. Rotate alice's key in trie: `trie.stub_rotate_member_key(&ALICE_ID, K2_alice)`.
3. The doc ACL (CRDT state) is **not modified** ŌĆö no CRDT operation needed.
4. After rotation: `effective_member_keys` returns `{K2_alice, K_bob}`.
5. `state.members(DOC_GID)` still lists `ALICE` by stable ID ŌĆö CRDT unchanged.

Result: PASS. Key rotation tracking works via the resolver; the CRDT is not
touched. This is the "no ACL mutation" property of the org principal.

## Escape hatches

1. **`members()` auto-resolves nested groups** ŌĆö `GroupCrdtState::members(doc_gid)`
   performs the nested-group walk internally. Phase 3's adapter does not need a
   custom walker for the `p2panda-auth` layer.

2. **`effective_member_keys` bridges CRDT IDs to live keys** ŌĆö `members()` returns
   stable `AuthMemberId` values; `effective_member_keys` translates these to
   `P2pMemberKey` values by calling the resolver. Members absent from the trie
   are silently skipped (correct: no stale-key entries).

3. **`GroupMember::Group` with non-Manage access only** ŌĆö groups cannot be added
   as `Manage`-level members. The org pseudo-group is limited to `Read` or `Write`
   access. This is a documented CRDT invariant, not a bug.

4. **Spaces layer bypassed** ŌĆö the L2 demonstration uses `p2panda-auth`'s CRDT
   directly, not `p2panda-spaces::Group::add`. The spaces integration layer has
   no nested-group API. Phase 3 must either (a) bypass spaces entirely for
   org-as-pseudo-group ACL management, or (b) fork p2panda-spaces to add
   `Group::add_nested(group_id, access)`.

5. **`P2pMemberKey: Hash` added to `spike-common`** ŌĆö `P2pMemberKey` gained
   `#[derive(Hash)]` to enable `HashSet<P2pMemberKey>` in `effective_member_keys`.
   `ed25519_dalek::VerifyingKey` already implements `Hash`, so this is a safe
   derive.

6. **`AuthMemberId` and `AuthOpId`: `Display` required** ŌĆö `GroupCrdtError` requires
   `ID: Display` and `OP: Display` (via `thiserror` + `Box<dyn Error>`). Both
   local newtypes implement `Display`. Same pattern as `DcgkaMemberId` in gate 3.

## Salvage path (apply project_p2panda_fork_policy)

Per policy: no `UpstreamPR`.

### p2panda-auth layer ŌĆö None
No salvage needed. `GroupMember::Group(ID)`, `GroupCrdt::process`, and
`members()` all work correctly for the org-as-pseudo-group pattern.

### p2panda-spaces layer ŌĆö Fork (same fork as gate 2)

The Hard at p2panda-spaces for gate 4 is a direct cascade of the gate 2 Hard:
both require the same local fork of `p2panda-spaces`.

**Fork plan** (same fork as gate 2, one additional change):

1. Fork p2panda-spaces locally (same fork already required by gate 2).
2. Add `pub use types::AuthGroupState;` to `lib.rs` (gate 2 requirement).
3. Add `Group::add_nested(group_id: GroupMemberId, access: Access<C>)` to
   `src/group.rs`, which constructs `GroupAction::Add { member: GroupMember::Group(group_id), access }`
   and calls the internal `process_local_control`. This exposes nested-group
   management at the spaces level.
4. `fix_path = Fork`, `fix_effort = Small` (a few lines of additional code in
   the already-required fork; the CRDT already supports nested groups).

**Alternative (no fork):** Use `p2panda-auth` CRDT directly, bypassing
`p2panda-spaces::Group::add` entirely for the org-as-pseudo-group case. This is
viable because `effective_member_keys` operates on the CRDT state snapshot, not
on the spaces runtime. The spaces integration would be notified of membership
changes via its event system. This is a higher-effort design pattern but keeps
the fork surface minimal. `fix_path = TraitImpl` in this variant.
