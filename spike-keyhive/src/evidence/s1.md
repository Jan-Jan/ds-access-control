## Concreteness probe

`Identifier`, `IndividualId`, `GroupId`, and `DocumentId` are all
concrete newtypes over `ed25519_dalek::VerifyingKey`. The principal
types are not generic in ID. The `Agent` enum's variants thread these
concrete types as the "principal" position; no generic substitution
of an opaque `MemberId` is possible at the type level.

This matches the inventory hypothesis: **gate 1 is Hard at the
principal-type layer**.

## Salvage path ‚Äî call-site adapter (`IdAdapter`)

`Identifier`'s field is `pub` ‚Äî spike code can construct one directly
from a `VerifyingKey`. `IndividualId`'s field is `pub` too ‚Äî and `From<VerifyingKey>` for `IndividualId` can be assembled in three lines:

```text
let vk: VerifyingKey = adapter.resolve(&resolver, &member_id).expect("in trie");
let individual_id = IndividualId(Identifier(vk));
```

`GroupId` and `DocumentId` are `pub(crate)` but carry `#[derive(From, Into)]`,
so they're constructible from `Identifier` via `GroupId::from(Identifier(vk))`.

The spike's `IdAdapter` (see `src/adapter.rs`) caches the mapping
`MemberId ‚Üí VerifyingKey` populated lazily from the `MemberKeyResolver`.
At every Keyhive call site that needs a principal, the spike code:
1. Resolves `MemberId` ‚Üí `VerifyingKey` via the adapter (which queries
   the trie if the entry isn't cached).
2. Wraps it in `Agent::Individual(IndividualId(Identifier(vk)), ...)`.
3. Calls `keyhive.add_member(agent, ...)`.

For revocation, `revoke_member` takes a bare `Identifier` ‚Äî the
adapter resolves `MemberId ‚Üí VerifyingKey ‚Üí Identifier` and passes it
directly.

## Rotation tracking

The delegation log inside Keyhive holds `Identifier` (= raw
`VerifyingKey`) verbatim. When a `MemberId` rotates in the trie, the
*existing* delegation entry still points at the old `VerifyingKey`.
The spike addresses this by:
1. `adapter.invalidate(&member_id)` ‚Äî drop the cached mapping.
2. On the next lookup, the adapter refreshes from the trie and the new
   `VerifyingKey` flows through to subsequent Keyhive calls.
3. To roll the existing delegation: the spike's gate-3 code invokes
   `keyhive.force_pcs_update(doc)` after the adapter is refreshed,
   which forces a CGKA rotation and a fresh delegation referencing the
   new key.

This is the same general pattern as p2panda's spike (stable-ID
indirection at the call site, plus a CGKA rotation hook on key
change), with two advantages over p2panda:
- `Identifier`'s `pub` field eliminates p2panda's `ActorId`
  `pub(crate)` friction (no fork needed for direct construction).
- The adapter is a single ~30-line module rather than a
  trait-impl-plus-fork spread across three subcrates.

## Integration finding (from running code, 2026-06-02)

`Individual::new(initial_op: KeyOp)` requires a signed `KeyOp`, not a
bare `VerifyingKey`. A `KeyOp` is the signed CRDT op publishing or
rotating an X25519 prekey ‚Äî it carries (ed25519 issuer, x25519 prekey,
signature). The high-level Keyhive-native add-member flow is:

1. Each member runs their own Keyhive instance; calls `keyhive.contact_card().await`
   to get a signed `ContactCard` (a serialisable wrapper around an
   initial `KeyOp`).
2. Peers ingest via `keyhive.receive_contact_card(&card).await`, which
   returns `Arc<Mutex<Individual>>`.
3. `add_member` accepts `Agent::Individual(id, individual_arc)` and
   atomically (a) records the delegation, (b) places the member in the
   document's BeeKEM tree, (c) advances the CGKA epoch.

The spike's [`l2_g1`](../../../tests/l2_g1.rs) test exercises this
high-level path end-to-end: two Keyhive instances exchange contact
cards through a `ContactCardForge`, and `transitive_members()` reports
both members after the add.

## Design pivot (2026-06-03) ‚Äî lazy CGKA / two-tier separation

The high-level `add_member` API conflates two things ODS wants to keep
separate: **ACL membership** (a slow-changing, trie-authoritative fact)
and **CGKA tree membership** (an ephemeral, online-only encryption
construct). The convenience API requires the new member's prekey at
add-time because it eagerly places them in the BeeKEM tree.

For ODS we commit to a **lazy-CGKA / two-tier** design:

| Tier | Source of truth | Cadence | Carries |
|------|-----------------|---------|---------|
| ACL  | Trie (on-chain) | Rare ‚Äî onboarding + rotation/revocation | `(MemberId, VerifyingKey)` |
| CGKA tree | Per-document BeeKEM state | Per-write, per-rotation | x25519 leaf keys (online members only) |

Mechanics:

1. **Grant.** Alice grants bob ACL access on doc D. The trie records
   the delegation against bob's `MemberId`/`VerifyingKey`. No KeyOp
   needed. No CGKA mutation. Bob may be offline.
2. **Come-online self-add.** When bob next syncs, his client sees the
   delegation, fetches doc D's current CGKA state, generates a fresh
   x25519 leaf, and commits a self-add to BeeKEM (signed by bob,
   verifiable as authorised by alice's delegation).
3. **History transfer.** Bob asks an already-authorised peer (alice)
   for the document's history. Alice has the plaintext at-rest;
   she retransmits it to bob wrapped under the current CGKA epoch
   (which bob is now in). No new privilege leak ‚Äî alice already had
   read access; she's transferring her view to a co-authorised member.
4. **Revocation.** Trie revokes bob's `MemberId`. Existing members
   commit a CGKA remove/update; bob's leaf is rotated out of the tree.
   Forward security holds via fresh `PcsKey` on the new epoch.

This sidesteps the prekey-publication question entirely. The trie
carries only the long-term `VerifyingKey` (rare updates, on-chain).
Prekeys live nowhere durable ‚Äî they are generated at self-add time and
consumed immediately in the CGKA tree.

### What this means for the library API

The two-tier model operates **below** `keyhive_core::Keyhive::add_member`
because that API eagerly places members in the CGKA tree. Phase 3
composes:

- **`keyhive_core` delegation log directly** for ACL grants. Delegations
  reference `Identifier(VerifyingKey)`, signed by the issuer. No
  `Individual` materialisation required at grant-time.
- **`beekem::Cgka::add(id, pk, signer)` directly** for CGKA placement,
  driven by the member's own client at come-online. The `pk` is the
  fresh x25519 leaf; `signer` is the member's signing key.

Both layers are reachable in the current pin ‚Äî `beekem` is a separate
sub-crate dependency in `Cargo.toml`. The spike's `s1_stable_id_acl.rs`
exercises the eager-add (high-level API) path for evidence completeness;
a follow-up L3 "lazy onboarding" scenario would demonstrate the
two-tier path end-to-end.

### Gap-matrix entry ‚Äî final

| Library | Gate | Flow | Principal | Severity | Subcrate | Fix path | Fix effort | Phase 3 effort |
|---|---|---|---|---|---|---|---|---|
| Keyhive | 1 | A | Member | Hard | keyhive_core | TraitImpl | Small | Small |

**Why Hard remains:** the principal type is concrete (`Identifier(VerifyingKey)`).
The salvage needs a call-site adapter ‚Äî it does not work zero-cost from
the type alone.

**Why phase3_effort is Small:** the trie carries the long-term
`VerifyingKey` only (no prekey/ContactCard publication). The call-site
`IdAdapter` (~30 lines) plus direct `beekem::Cgka` composition for CGKA
placement is bounded, contained work.

**Why fix_effort is Small:** the spike's eager-add evidence in
`s1_stable_id_acl.rs` (~70 lines including `ContactCardForge`) confirms
both layers compose. The two-tier production path uses strictly fewer
moving parts ‚Äî direct delegation construction + `beekem::Cgka::add` ‚Äî
not more.

### Comparison vs p2panda

Both libraries' high-level "add member" APIs conflate ACL+CGKA and
require the new member's prekey/KeyBundle at add-time. p2panda's
`manager.add_member(member_id, key_bundle)` is the symmetric case to
Keyhive's `keyhive.add_member(Agent::Individual, ...)`. To get the
lazy-CGKA model, both libraries require descending below the convenience
API: p2panda to `Dcgka::*` + `GroupCrdt` auth-layer directly; Keyhive to
`beekem::Cgka` + `keyhive_core` delegation log directly.

For the burden estimate this means: **Keyhive's gate-1 burden returns to
parity with p2panda's** (both Small/Small). The original (pre-spike)
analytical prediction was correct; the running-code milestone surfaced
a high-level-API observation that is informative but not architectural.
