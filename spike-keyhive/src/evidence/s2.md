# Gate 2 evidence — Keyhive membership-op interception

## Verified API surface at pin `a2876f3c`

```text
// keyhive_core/src/listener/membership.rs
pub trait MembershipListener<F: FutureForm, S: AsyncSigner<F>, T: ContentRef>:
    PrekeyListener<F> + CgkaListener<F>
{
    fn on_delegation<'a>(&'a self, data: &'a Arc<Signed<Delegation<F, S, T, Self>>>) -> F::Future<'a, ()>;
    fn on_revocation<'a>(&'a self, data: &'a Arc<Signed<Revocation<F, S, T, Self>>>) -> F::Future<'a, ()>;
}

// keyhive_core/src/store/ciphertext.rs
pub trait CiphertextStore<F: FutureForm, Cr: ContentRef, T> {
    type GetCiphertextError;
    type MarkDecryptedError;
    fn get_ciphertext(&self, key: Cr) -> F::Future<'_, Result<Option<...>, Self::GetCiphertextError>>;
    fn mark_decrypted(&self, key: Cr) -> F::Future<'_, Result<(), Self::MarkDecryptedError>>;
    /* ... */
}

// keyhive_core/src/keyhive.rs (relevant generics)
pub struct Keyhive<F, S, T, P, C, L, R>
where
    C: CiphertextStore<F, T, P> + CiphertextStoreExt<F, T, P> + Clone,
    L: MembershipListener<F, S, T>,
    /* DelegationStore<F, S, T, L> and RevocationStore<F, S, T, L> are
       concrete fields — NOT generic, NOT configurable */
```

## Probe results

**MembershipListener is post-fact, not an intercept seam.**
`on_delegation` and `on_revocation` return `F::Future<'a, ()>` —
unit type. They are fired AFTER the internal store writes have
already completed. An impl that wanted to "refuse" an operation
has no return path; the operation has already committed when the
listener fires. The hypothesis ("Soft via custom CiphertextStore /
DelegationStore impls") only partially applies — `DelegationStore`
and `RevocationStore` are concrete struct fields, not trait-bound
generics. They cannot be replaced by a custom impl.

**CiphertextStore IS swappable but is the wrong intercept layer.**
The `CiphertextStore<F, Cr, T>` trait CAN refuse writes by returning
`Err(Self::MarkDecryptedError)`, but it only governs *encrypted
content* persistence — not membership delegations or revocations.
Using it to intercept membership mutations is type-incorrect:
membership state never flows through `CiphertextStore`.

## Salvage path — call-site wrapper

Since the library-internal stores and listeners cannot refuse
operations, the spike's gate 2 substitution is at the **call-site
wrapper layer**: the spike's `KeyhiveWrapper<R: MemberKeyResolver>`
holds the `Keyhive` instance in a private field. Only the wrapper's
own methods (`grant_member`, `revoke_member`) — driven by the
trie-resolver code path — invoke `keyhive.add_member` and
`keyhive.revoke_member`. Application code that gets a reference to
the wrapper has no access to the underlying `Keyhive`.

This is a *containment* intercept, not a *refusal* intercept: it
ensures only the trie-driven path can mutate state, but at the cost
of relying on the wrapper not leaking the inner `Keyhive` handle.

For Phase 3, this is the same pattern any production codebase would
use anyway (the application doesn't expose CGKA primitives directly
to user code), so the practical fix_effort is Small.

**Why Soft not Hard:** containment-via-wrapper is a viable
intercept, even though it differs from the "store-level refusal"
hypothesis. The substitution works; the implementation shape is
different from p2panda's `BlockingGroups<Inner>` wrapper but the
overall effect is identical.
**Why fix_effort Small:** the wrapper is a thin newtype around
`Arc<Keyhive<...>>` exposing only the trie-driven entry points.

## Comparison vs p2panda

p2panda recorded `Hard / Fork Small` for gate 2 because
`AuthGroupState<C>` is in a private module (`pub use` required).
Keyhive's gate 2 is `Soft / TraitImpl Small` because there's no
type-system blocker — only the architectural choice that the
internal store is not configurable. Net: Keyhive's gate 2 is
slightly cleaner.
