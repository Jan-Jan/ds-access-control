# Gate 4 evidence — Keyhive organisation-as-pseudo-group

## Verified API surface at pin `a2876f3c`

```text
// keyhive_core/src/principal/agent.rs
pub enum Agent<F, S, T, L> {
    Active(IndividualId, Arc<Mutex<Active<F, S, T, L>>>),
    Individual(IndividualId, Arc<Mutex<Individual>>),
    Group(GroupId, Arc<Mutex<Group<F, S, T, L>>>),       // <-- first-class
    Document(DocumentId, Arc<Mutex<Document<F, S, T, L>>>),
}

// keyhive_core/src/principal/group.rs (or .../group/mod.rs)
impl<F, S, T, L> Group<F, S, T, L> {
    pub async fn transitive_members(&self)

}

// keyhive_core/src/principal/document.rs (or .../document/mod.rs)
impl<F, S, T, L> Document<F, S, T, L> {
    pub async fn transitive_members(&self)
        -> HashMap<Identifier, (Agent<F, S, T, L>, Access)>;
}

// keyhive_core/src/keyhive.rs
pub async fn generate_group(
    &self,

    &self,

    resource: &Membered<F, S, T, L>,
```

## Flow A — Org granted access to a doc

`Agent::Group(GroupId, Arc<Mutex<Group<...>>>)` is a first-class
variant of `Agent`. `Keyhive::add_member(to_add: Agent<...>, ...)`
accepts ANY variant — passing `Agent::Group(org_group_id, org_group)`
makes the org group a member of the document with the chosen access
level. **Identical call signature to adding an individual.**

This is a strong differentiator vs p2panda: in p2panda-spaces,
`Space::add(member: ActorId, access)` accepts only the individual
`ActorId` newtype — adding a `GroupMember::Group(org_id)` requires
bypassing the spaces layer entirely and writing ACL entries at the
p2panda-auth layer through a forked `pub use`.

The spike's gate-4 wrapper:
```text
pub async fn grant_org<R, F, S, T, L>(
    keyhive: &Keyhive<F, S, T, ..., L>,
    org_group: Arc<Mutex<Group<F, S, T, L>>>,
    org_group_id: GroupId,
    doc: &Membered<F, S, T, L>,
    access: Access,
) -> Result<AddMemberUpdate<F, S, T, L>, AddMemberError> {
    keyhive.add_member(
        Agent::Group(org_group_id, org_group),
        doc, access, &[],
    ).await
}
```

`Document::transitive_members()` resolves the org group automatically:
the returned `HashMap<Identifier, (Agent, Access)>` contains every
individual member of the org, mapped to their derived access.

## Flow C — Rotation cascade

When an individual member rotates their `p2p_member_key` in the trie
while they're a member of the org:
1. `adapter.invalidate(&member_id)`.
2. The spike's trie-change observer must know *which documents have
   the org as a member* — this is the "reverse lookup" gap also
   present in gate 5.
3. For each such doc, call `keyhive.force_pcs_update(doc)`.
4. The new PCS key forward-secures the doc.

For Flow C-on-org-member-removal (alice is removed from the org):
1. `keyhive.revoke_member(alice_identifier, false, &org_group_membered)`.
2. The org group's `MembershipListener::on_revocation` fires with the
   signed revocation op.
3. A spike-layer dispatcher reads the op, walks the spike's reverse
   index "org → docs", and calls `force_pcs_update` on each doc.
4. Subsequent `transitive_members()` on those docs no longer includes
   alice's `Identifier`; CGKA secrets are rotated.

## Reverse-index gap (shared with gate 5)

Both Flow C variants depend on a "given org, list docs where org is a
member" query. Keyhive itself does not expose this index — it would be
the inverse of the delegation graph. The spike maintains this index
in its own wrapper state, populated on every `grant_org` call. Phase 3
needs either (a) a Keyhive API for this query, or (b) acceptance that
the wrapper owns this index.

## Gap-matrix entries

| Keyhive | 4 | A | Org | Soft | (none) | TraitImpl | Small | Small |
| Keyhive | 4 | C | Org | Soft | (none) | TraitImpl | Small | Medium |

**Why Soft and not None for Flow A:** still need the wrapper that
constructs `Agent::Group(...)` and tracks the reverse index.
**Why Soft and not Hard for Flow C:** the cascade is straightforward
once the reverse index is in place; no architectural blocker.
**Why phase3 Medium for Flow C:** the reverse-index maintenance adds
non-trivial code paths to keep consistent.

## Comparison vs p2panda

p2panda recorded `Hard / Fork Small` for both Flow A and Flow C under
the Org principal — Hard because `Group::add(member: ActorId)` at the
spaces layer doesn't accept nested groups, so the salvage requires a
`pub use` patch to access private types and bypass the spaces-layer
gatekeeper.

Keyhive records `Soft / TraitImpl Small` for the same flows: no fork
needed, the substitution lives entirely in spike code, and the
mechanism (`Agent::Group(...)`) is upstream-native.

**This is the strongest single Keyhive-vs-p2panda differentiator
in the Phase 1.d data.**
