## Flow B — CGKA compute over resolver-supplied keys

`Cgka::add(id, pk, signer)` takes the leaf encryption key as a direct
argument. This is the clean key-injection seam Gate 3 needs — strictly
better than p2panda's `IdentityRegistry` trait-impl pattern, because
no generic-trait threading is required.

The spike's gate-3 helper resolves `MemberId → P2pMemberKey` via the
trie, packs as a `ShareKey`, and feeds it directly to `Cgka::add`:

```text
let new_vk = adapter.resolve(&resolver, &member_id).expect("in trie");
let share_key: ShareKey = ShareKey::from(new_vk.to_bytes()); // or via keyhive_crypto
cgka.add(member_id_as_beekem_id, share_key, &signer).await?;
```

Or, at the higher Keyhive layer, by calling `add_member` after the
adapter is refreshed.

## Flow C — Recompute on rotation

`Keyhive::force_pcs_update(doc)` is the rotation entry point at the
keyhive_core layer. It triggers a `Cgka::update(...)` internally,
generating a new `PcsKey` and signed `CgkaOperation`. The spike's
trie-change observer:
1. Calls `adapter.invalidate(&member_id)`.
2. Calls `keyhive.force_pcs_update(doc).await`.
3. The new PCS key forward-secures the doc against any old key
   material; new content is encrypted under the fresh PCS key.

The BeeKEM tree's O(log n) cost per rotation is the headline asymptotic
advantage over p2panda's DCGKA O(n). For an org of 100 members this is
~7 ops vs ~100 ops per rotation; for 10,000 members it's ~14 vs 10,000.
This compounds significantly under frequent revocation churn.

## Comparison vs p2panda

p2panda recorded `Soft / TraitImpl Small` for both Flow B and Flow C,
matching Keyhive. The key differences:
- **Mechanism:** p2panda threads a custom `IdentityRegistry` through
  DCGKA's `PKI` generic. Keyhive injects the leaf key directly via
  `Cgka::add`. Keyhive's is mechanically simpler.
- **Asymptotic cost:** Keyhive's BeeKEM is O(log n); p2panda's DCGKA is
  O(n). For ODS-scale orgs (10s to 1000s of members) this favours
  Keyhive substantially.
- **Integration finding:** p2panda's L3 revocation uncovered that
  `Dcgka::update` sends DMs to all DGM members including the removed
  one — production code must dispatch to `Dcgka::remove` instead.
  Keyhive's API distinguishes these cleanly: `Cgka::remove(id, signer)`
  for removal-rotation; `Cgka::update(new_pk, ...)` for refresh-rotation.
  No integration finding for Keyhive here.
