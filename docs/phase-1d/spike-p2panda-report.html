1. Summary

The spike confirms p2panda is a viable substrate for ODS, with caveats that are bounded and locally fixable. All six substitutions the design requires can be reproduced in working code without modifying p2panda's source — except for one narrow visibility issue at p2panda-spaces, and (if WASM is required) one transport-layer dependency at p2panda-sync.

Without WASM requirement

Burden 20

4 Hard rows, 6 Soft rows. One small fork covers 3 of the 4 Hard rows (single pub use patch in p2panda-spaces). The remaining Hard is a clean TraitImpl already implemented in the spike.

With WASM requirement

Burden 32

Adds gate 0 (compile to wasm32-unknown-unknown). 5 sub-crates pass; p2panda-sync requires a Large fork — swap tokio for tokio_with_wasm. This is the largest single item of work.

Test coverage

129 / 129

L1 (per-sub-crate), L2 (per-gate integrated), and L3 (per-scenario end-to-end) all green. Three scenarios from the design's §Data flow are exercised through the composed substitutions.

Substitution shape

~600 LOC

Five gate modules in spike-p2panda/src/sN_*.rs totalling about 600 lines. All gate code uses public p2panda APIs at the pinned revision; no upstream modifications.

Per-library qualification. p2panda is qualified as a viable phase-3 substrate. The decision is straightforward without the WASM requirement; with WASM it carries one Large fork of p2panda-sync, which is real but not architecturally blocking.

Head-to-head outcome (2026-06-03). The consolidated Phase 1.d decision (decision.md) picks Keyhive as the primary substrate per the hard-blocker rule (Keyhive 1 Hard vs p2panda 5 Hard) confirmed by override-on-cost (burden 22 vs 32). p2panda is retained as the documented salvage alternative if Keyhive fails its maturity gate; the per-library findings in this report constitute that salvage plan's evidence base.

The ODS lazy-CGKA / two-tier design commitment (lazy-cgka-results.md) applies symmetrically to p2panda: a phase-3 wiring on p2panda would also descend below p2panda_spaces::Manager::add_member to GroupCrdt + Dcgka::* directly so the trie carries only long-term keys, not KeyBundles at add-time. Both libraries support the lower-level composition; the burden estimates in this report assume that descent is taken.

2. Methodology and ground rules

Six gates evaluated

The spike exercises the six gates defined in the design document:

GateWhat it testsSpec mapping
0 Each p2panda sub-crate compiles to wasm32-unknown-unknown under no_std + alloc + serde Workspace convention (matches org-members)
1 ACL entries reference stable MemberId, not raw VerifyingKey; resolver materialises the key at use time §Key changes #1; phase 1.d (a)
2 Library-native membership-mutation ops (add/remove) are disabled or intercepted; the trie is the sole write authority §Key changes #3; phase 1.d (c)
3 (D)CGKA / member-as-a-group key rotation is driven by trie change, not by an internal library cache §Key changes #4; phase 1.d (b)
4 Organisation-as-pseudo-group is expressible as an ACL principal — any current member acts as the org §Key changes #2 (scope expansion vs phase 1.d wording)
5 Peer-to-peer connection policy consults the trie at session-establish time and on trie change §Key changes #5 (scope expansion)

Three test levels

Per the design's layered test pyramid, every gate is exercised at three levels:

  • L1 — per sub-crate. Tests the substitution against the smallest p2panda crate that owns the primitive. Locates failures to a specific sub-crate.
  • L2 — per gate, integrated. Exercises the substitution through the library's normal composition.
  • L3 — per scenario. End-to-end fixtures (revocation, gating, org-pseudo-group) composing multiple gates.

Severity classifications

SeverityMeaning
None The substitution composes cleanly via existing public p2panda APIs.
Soft Implementable with an adapter or wrapper that conforms to a public trait. Phase-3 friction is documented but bounded.
Hard The library's relevant primitive is private or sealed against the extension point needed. A salvage path (fork, replace, or trait-impl) is required.

Salvage path policy

Project policy: No upstream PRs to p2panda are proposed for Phase 1.d. The salvage path for every Hard row is either a local fork or a trait implementation in spike-p2panda. Upstream contribution will be reconsidered after the spike demonstrates value (project_p2panda_fork_policy, 2026-05-28).

Effort accounting

The burden number combines two estimates per row:

  • Phase 3 effort — work to integrate the substitution into a production stack.
  • Fix effort — work to implement the salvage (fork patch, trait impl, replacement).

Effort weights: Small = 1, Medium = 3, Large = 9, XL = 81 (super-linear; penalises one big gap more than several small ones).

3. Part 1 — Findings without WASM requirement

If WASM compatibility is not a hard requirement for Phase 3, gate 0 is dropped. The remaining gates 1–5 yield 4 Hard + 6 Soft rows, total burden 20. Three of the four Hard rows share a single root cause and a single one-line fork patch in p2panda-spaces.

Gate 1 — Stable-ID ACL with trie-lookup resolver

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
A (delegation)MemberHardp2panda-spacesTraitImplSmall

What was found. Severity is split between the two layers:

  • At p2panda-auth: None. GroupCrdt<ID, …> is fully generic over ID: IdentityHandle; MemberId satisfies the bound; ACL serialises opaque ID bytes (CBOR round-trip confirms [0xa1; 32] patterns, no ed25519 key material).
  • At p2panda-spaces: Hard. Group::add / Space::add accept only ActorId (a newtype over VerifyingKey) concretely; Manager<ID, …>'s ID generic is the space identifier, not the member identifier.

Salvage already implemented in the spike (spike-p2panda/src/s1_stable_id_acl.rs, ~40 functional lines):

  • materialise_actor_id(resolver, principal) -> Result<ActorId, _> — the spaces-layer wrapper. Resolves a Principal to a freshly-materialised ActorId at call time. If the resolver's mapping changes, the next call returns the new key.
  • ResolverPki<R> — implements p2panda-encryption's IdentityRegistry<MemberId, ResolverPki<R>>. This is the encryption-layer adapter, reused by gate 3.

Positive surprise. From<VerifyingKey> for ActorId is public — the wrapper is one call, not a workaround.

Escape hatch. IdentityRegistry::identity_key is a static method (no &self). The resolver is threaded via the Y type parameter, instantiated as ResolverPki<R>. Unconventional but valid; alternative is to fork p2panda-encryption to change the receiver to &self (deferred under the fork-locally policy).

Evidence: spike-p2panda/tests/l1_p2panda_auth.rs, spike-p2panda/tests/l1_p2panda_spaces.rs, spike-p2panda/tests/l2_g1.rs, spike-p2panda/src/evidence/s1.md.

Gate 2 — Library-native membership-mutation interception

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
D (op interception)N/AHardp2panda-spacesForkSmall

What was found. Again split by layer:

  • At p2panda-auth: None. The Groups trait is public and externally implementable. BlockingGroups<Inner> (in spike-p2panda/src/s2_membership_intercept.rs) wraps any Groups impl and returns Err(MutationBlocked) on all six mutation methods.
  • At p2panda-spaces: Hard. The AuthStore<C> trait is exported as public, but its parameter type AuthGroupState<C> uses AuthMessage<C> from a private module. The trait is externally visible but not externally implementable. Compile error E0603 confirmed.

Salvage: a one-line fork of p2panda-spaces. Add:

pub use types::AuthGroupState;

to expose the type. This single patch unblocks gates 2, 4A, and 4C (same root cause).

Evidence: spike-p2panda/tests/l1_p2panda_auth_intercept.rs, spike-p2panda/tests/l2_g2.rs, spike-p2panda/src/evidence/s2.md.

Gate 3 — (D)CGKA recompute on trie key change

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
B (compute)MemberSoft—TraitImplSmall
C (recompute on rotation)MemberSoft—TraitImplSmall

What was found. Full DCGKA flow demonstrated:

  • Flow B (compute). Dcgka::create + Dcgka::process succeed with KeyRegistry<DcgkaMemberId> as the PKI generic. Alice and Bob derive an identical GroupSecret.
  • Flow C (recompute on rotation). trigger_recompute (in s3_cgka_rotation.rs) drives Dcgka::update after re-populating the KeyRegistry from the resolver. Epoch 0 ≠ epoch 1; the new shared secret cannot be derived from the pre-rotation key.

Important architectural finding. ResolverPki<R> (from gate 1) cannot serve directly as the PKI generic, because the DCGKA state requires PKI to be Serialize + Deserialize and to hold pre-key bundles. The correct integration pattern is:

  1. Use KeyRegistry<MemberId> as the concrete PKI state.
  2. Populate it from the resolver at group setup and at rotation time.
  3. Drive Dcgka::update via trigger_recompute after re-population.

Escape hatches. DcgkaMemberId newtype (orphan rule); DcgkaError requires ID: Display; test_utils feature needed for KeyManager::init_and_generate_prekey in test fixtures.

Evidence: spike-p2panda/tests/l1_p2panda_encryption.rs, spike-p2panda/tests/l2_g3.rs, spike-p2panda/src/evidence/s3.md.

Gate 4 — Organisation-as-pseudo-group principal

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
A (delegation)OrgHardp2panda-spacesForkSmall
C (recompute on rotation)OrgHardp2panda-spacesForkSmall

What was found.

  • At p2panda-auth: None (positive surprise). GroupMember::Group(ID) is accepted by GroupCrdt::process; members() auto-resolves nested groups — no manual walker required in phase 3.
  • At p2panda-spaces: Hard. Same root cause as gate 2 (private type). The spike implements OrgPseudoGroupAdapter and effective_member_keys at the auth layer, bypassing the spaces-layer API for org-keyed flows.

Documented constraint. GroupMember::Group with Access::manage() is rejected by the CRDT (ManagerGroupsNotAllowed). Org pseudo-group entries must use Read or Write access; managerial actions stay with Individual members. This matches the ODS design (admin status is on-chain, not in the local-first ACL).

Salvage. Same one-line fork as gate 2 (covered transitively).

Evidence: spike-p2panda/tests/l1_p2panda_auth.rs (tests 4–6), spike-p2panda/tests/l2_g4.rs, spike-p2panda/src/evidence/s4.md.

Gate 5 — Peer-to-peer connection policy

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
E1 (accept/reject)MemberSoft—TraitImplSmall
E2 (accept/reject)OrgSoft—TraitImplSmall
F1 (termination on revocation)MemberSoft—TraitImplSmall
F2 (termination on org rotation)OrgSoft—TraitImplSmall

What was found. PolicyManager<R, F> (in s5_p2p_policy.rs) wraps the session-establish flow:

  • SessionConfig::remote: VerifyingKey is available at session-open time — this is the intercept hook.
  • policy_check(peer_key, doc_id) consults the resolver via a closure that maps VerifyingKey → MemberId.
  • recheck_open_sessions() walks open sessions on trie change and flags those whose remote device is no longer authorised.

All four sub-flows pass with explicit acceptance / rejection / termination assertions.

Discovered gap in spike-common. The MemberKeyResolver trait has no reverse lookup (VerifyingKey → MemberId). The spike works around this with a closure parameter on PolicyManager. Phase 3 should add this method to the resolver trait (or formalise a KeyIndex companion).

Discovered gap — push notification. recheck_open_sessions() is a manual invocation. The StubTrie is pull-only; phase 3 needs either a push from the trie or a polling strategy.

Pre-open vs post-open termination. The policy check accepts or rejects at the session-establish moment (pre-open). For termination on trie change, the spike flags sessions for close after the trie observer fires; there is a small post-open window before the close actually propagates to the wire. The window is recorded as a phase-3 risk note.

Evidence: spike-p2panda/tests/l1_p2panda_sync.rs, spike-p2panda/tests/l2_g5.rs, spike-p2panda/src/evidence/s5.md.

5. Integration findings (cross-cutting)

These are findings discovered during the spike that affect more than one gate and need explicit phase-3 follow-through.

5.1 Dcgka::remove vs Dcgka::update for revocation

Gate 3's trigger_recompute wraps Dcgka::update. This is correct for the key-rotation flow (Flow C). It is the wrong operation for membership revocation: Dcgka::update sends direct messages to every member listed in the DGM, including the member being removed, breaking forward-security.

The correct DCGKA operation for revocation is Dcgka::remove, which filters out the revoked member from the direct-message fan-out. The L3 revocation test uses Dcgka::remove directly.

Phase 3 implication: the trie-change observer must distinguish rotation events from revocation events, and dispatch to the appropriate DCGKA operation. Documented in spike-p2panda/src/evidence/s3.md.

5.2 Reverse lookup gap in MemberKeyResolver

Gate 5 requires mapping VerifyingKey → MemberId at session-establish time, but the foundation MemberKeyResolver trait only goes the other way. The spike resolves this with a closure injected into PolicyManager. Phase 3 should formalise a find_member_by_device(VerifyingKey) method on the resolver trait (or a sibling KeyIndex).

5.3 Trie push notification missing

The StubTrie in spike-common is pull-only. recheck_open_sessions() must be invoked manually. Phase 3 needs either a push channel from the trie (e.g. a tokio::sync::watch) or a polling strategy in the policy manager.

5.4 Pre-1.0 API instability

The pinned p2panda revision (41559b0, 2026-05-20) was followed two days later by API-renaming commits in p2panda-spaces. The spike's findings are valid for the pinned revision; any phase-3 work must either hold the pin or re-run the L1 battery against a newer commit.

8. Recommendations for Phase 3

If WASM is not required

  1. Fork p2panda-spaces with one pub use patch. Small effort. Unblocks gates 2, 4A, 4C.
  2. Adopt the spike's TraitImpl code as Phase 3 seed code. The ~600 LOC across s1..s5_*.rs is production-quality scaffolding.
  3. Extend MemberKeyResolver with the reverse-lookup method discovered in gate 5.
  4. Wire trie push notifications for the policy manager.
  5. Separate Dcgka::remove from Dcgka::update in the trie-change observer (substitution #4).

If WASM is required

Add to the above:

  1. Fork p2panda-sync to swap tokio for tokio_with_wasm. Large effort. Plan for ~3 weeks plus integration testing on a real WASM consumer (browser PWA).
  2. Pin the getrandom family at the consumer level per gate-0 evidence. Two pins needed (0.2 with js and 0.3 with wasm_js).

Independent of WASM

  • Hold the p2panda pin at 41559b0 through Phase 3. Any newer commit must re-run the L1 battery — the upstream is pre-1.0 and APIs rename.
  • Schedule an independent security audit after Phase 3 integration, per the parent design's §Addendum. The p2panda-encryption audit by Radically Open Security is a head-start; the rest of the integrated stack remains unaudited.
