1. Summary

The spike confirms Keyhive is a viable substrate for ODS with materially less fork burden than p2panda. All six substitutions the design requires can be reproduced against Keyhive's public APIs at the pinned revision without modifying upstream source. Only one Hard row appears — the principal-ID concreteness — and its salvage is a ~30-line call-site adapter inside spike-keyhive, with no fork required.

Without WASM requirement

Burden 22

1 Hard row, 9 Soft rows. The Hard is gate 1 (call-site IdAdapter already implementable in spike code). No fork needed for any gate.

With WASM requirement

Burden 22

Adds gate 0 (WASM/no_std). All three Keyhive sub-crates compile cleanly to wasm32-unknown-unknown — gate 0 scores None and contributes zero to the burden.

Comparison vs p2panda

−31%

Keyhive's WASM burden is 22 vs p2panda's 32 (−10 points, −31%). The largest single saving is gate 0 (zero forks vs one Large fork) and the gate-4 org-as-pseudo-group (Soft vs Hard).

Substitution shape

No forks

Every Keyhive gap is salvageable via TraitImpl: the spike's IdAdapter, a containment wrapper, custom listeners, and a session-policy stub. Pure consumer-side code.

Headline finding. Keyhive's principal model (Agent::Group(...) as a first-class variant of Agent) and CGKA primitive (beekem::Cgka::add(id, pk, signer)) accept the spike's substitutions natively — no upstream gatekeeper. This produces a fork-free integration path that p2panda cannot match at the pinned revision.

Caveat — alpha stability. Keyhive is workspace version 0.0.0-alpha.3 and explicitly unaudited. The library is cryptographically rigorous (BeeKEM is published research) but the public Rust API surface is pre-1.0 and may change. Phase 3 must re-pin and re-run the L1 battery before any production commit.

Running-code update (2026-06-03). 24 tests pass in spike-keyhive/ covering every gate (18 lib + 2 L2 + 4 L3 scenarios: revocation, gating, org_pseudo_group, lazy_onboarding). The ODS architecture commits to a lazy-CGKA / two-tier design that operates below Keyhive's eager-add convenience API: the trie publishes long-term VerifyingKey only; CGKA placement is deferred to the new member's own client at come-online. The L3 lazy_onboarding test validates this end-to-end on first run. Full design narrative: lazy-cgka-results.md.

2. Methodology and ground rules

Spike scope

The spike-keyhive effort started API-evidence-driven: each gate finding is grounded in the exact public-API surface of Keyhive at the pinned SHA, cited file-by-file in spike-keyhive/src/evidence/sN.md.

A running-code milestone (2026-06-02) then converted the analytical claims into 24 passing tests in spike-keyhive/ (18 lib + 2 L2 + 4 L3). The running-code path surfaced a high-level-API observation (Keyhive::add_member requires a signed KeyOp/ContactCard at add-time) which in turn drove an ODS design commitment on 2026-06-03: the lazy-CGKA / two-tier separation (see §5 cross-cutting and the focused results in lazy-cgka-results.md). Production composes keyhive_core delegation + beekem::Cgka directly, bypassing the eager-add convenience path; the L3 lazy_onboarding test validates the design's semantics on first run (no integration finding surfaced).

Phase 3 should re-pin to a tagged Keyhive release and re-run the L1+L2+L3 battery against the new pin before production commit. The current evidence is sufficient to score every gate with confidence at the alpha pin.

Six gates evaluated

The spike exercises the six gates defined in the design document:

GateWhat it testsSpec mapping
0 Each Keyhive sub-crate compiles to wasm32-unknown-unknown Workspace convention
1 ACL entries reference stable MemberId, not raw VerifyingKey; resolver materialises the key at use time §Key changes #1; phase 1.d (a)
2 Library-native membership-mutation ops are disabled or intercepted; the trie is the sole write authority §Key changes #3; phase 1.d (c)

Severity, fix-path, and effort schema

Same as the p2panda report (see §2 there for the rubric). Recapped briefly:

  • None — substitution composes via existing public APIs; no salvage needed.
  • Soft — implementable with an adapter or wrapper conforming to a public trait/seam.
  • Hard — primitive is private or sealed against the extension point needed; salvage requires fork, replace, or trait-impl.

Effort weights: Small=1, Medium=3, Large=9, XL=81 (super-linear).

Fork-locally policy applies

Same policy as spike-p2panda: no upstream PRs to Keyhive proposed at this phase. Salvage paths are TraitImpl or Fork only. (No Keyhive gate requires a fork — every salvage is a TraitImpl in spike-keyhive code.)

3. Part 1 — Findings without WASM requirement

If WASM compatibility is not a hard requirement for Phase 3, gate 0 is dropped. The remaining gates 1–5 yield 1 Hard + 9 Soft rows, total burden 22. The single Hard row has an in-spike TraitImpl salvage; no fork needed anywhere.

Gate 1 — Stable-ID ACL with trie-lookup resolver

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
A (delegation)MemberHardkeyhive_coreTraitImplSmall

What was found. Keyhive's principal types are concrete newtypes over ed25519_dalek::VerifyingKey:

// keyhive_core/src/principal/identifier.rs
pub struct Identifier(pub ed25519_dalek::VerifyingKey);

// keyhive_core/src/principal/individual/id.rs
pub struct IndividualId(pub Identifier);

// keyhive_core/src/principal/group/id.rs  (derive(From, Into))
pub struct GroupId(pub(crate) Identifier);

// keyhive_core/src/principal/document/id.rs  (derive(From, Into))
pub struct DocumentId(pub(crate) Identifier);

// keyhive_core/src/keyhive.rs
pub async fn add_member(
    &self,
    to_add: Agent<F, S, T, L>,        // accepts Individual, Group, Document, Active
    resource: &Membered<F, S, T, L>,
    can: Access,
    other_relevant_docs: &[Arc<Mutex<Document<F, S, T, L>>>],
) -> Result<AddMemberUpdate<F, S, T, L>, AddMemberError>;

None of these types is generic in ID — the principal model is fixed at ed25519. The spike's salvage is the call-site IdAdapter in spike-keyhive/src/adapter.rs:

pub struct IdAdapter {
    mapping: Arc<Mutex<HashMap<MemberId, VerifyingKey>>>,
}
impl IdAdapter {
    pub fn resolve<R: MemberKeyResolver>(&self, r: &R, id: &MemberId)
        -> Option<VerifyingKey> { /* cache + trie lookup */ }
    pub fn invalidate(&self, id: &MemberId);
    pub fn member_id_for(&self, vk: &VerifyingKey) -> Option<MemberId>;
}

At every Keyhive call site that needs a principal, the spike resolves MemberId → VerifyingKey via the adapter, wraps in Agent::Individual(IndividualId(Identifier(vk)), ...), and calls keyhive.add_member(agent, ...).

Positive structural detail. Identifier's field is pub — spike code constructs it directly, no factory required. GroupId / DocumentId use #[derive(From, Into)] so cross-construction from Identifier is one call. This is materially cleaner than p2panda's ActorId(pub(crate) VerifyingKey) which required us to discover From<VerifyingKey> for ActorId as an alternative.

Rotation behaviour. The delegation log holds the raw VerifyingKey verbatim — when a MemberId rotates in the trie, existing delegations point at the old key. The spike resolves this by composing the adapter with gate 3: on a trie rotation event, the adapter is invalidated for the rotating MemberId, and keyhive.force_pcs_update(doc) is called to forward-secure the doc with the new key.

Why Hard (not Soft). Severity is rated against the data-model semantics, not the salvage shape. The principal type is concrete AND the delegation log holds raw keys, so without a rotation hook a MemberId rotation produces a stale delegation. The salvage works, but the underlying constraint is real, so Hard.

Running-code follow-up & design pivot (2026-06-03). Implementing gate 1 as running Rust code surfaced that the high-level Keyhive::add_member(Agent::Individual, ...) path requires an Individual, which is constructed via Individual::new(KeyOp) from a signed prekey op (a ContactCard). An interim revision (briefly) raised phase3_effort to Medium because that would have required the trie to publish or sign ContactCards. ODS instead commits to a lazy-CGKA / two-tier design that operates below this convenience API — the trie publishes only the long-term VerifyingKey; ACL grant is a trie record; CGKA placement is performed by the new member's own client at come-online via beekem::Cgka::add directly. phase3_effort reverts to Small. The L3 lazy_onboarding test validates the design's semantics (BeeKEM forward security in the new-member direction; history transfer via plaintext re-transmission under the new epoch). Full design narrative: lazy-cgka-results.md.

Evidence: spike-keyhive/src/evidence/s1.md, spike-keyhive/src/adapter.rs, spike-keyhive/src/s1_stable_id_acl.rs, spike-keyhive/tests/l2_g1.rs, spike-keyhive/tests/l3_lazy_onboarding.rs.

Gate 2 — Library-native membership-mutation interception

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
D (op interception)N/ASoftkeyhive_coreTraitImplSmall

What was found. Two intercept strategies tried; one discarded:

  1. MembershipListener intercept — INFEASIBLE. on_delegation and on_revocation return F::Future<'a, ()> (unit type). They are fired AFTER the internal store writes complete. An impl cannot refuse the operation — the state is already mutated by the time the listener runs.
  2. Store-level intercept — TYPE-INCORRECT. CiphertextStore is swappable as a generic and CAN refuse writes by returning Err — but it governs only encrypted content, not membership. DelegationStore and RevocationStore are concrete struct fields of Keyhive, not generic — no custom impl possible.
  3. Containment-via-wrapper — VIABLE. The spike's gate-2 pattern keeps Arc<Keyhive<...>> in a private field of KeyhiveWrapper; only the wrapper's trie-driven entry points (grant_member, revoke_member) can invoke the underlying add_member/revoke_member.

Net. Containment-via-wrapper is functionally identical to p2panda's BlockingGroups<Inner> but is the only viable intercept here. The hypothesis (Soft via store-trait) was directionally right; the precise mechanism is different. This is the same pattern any production codebase would use anyway.

Evidence: spike-keyhive/src/evidence/s2.md.

Gate 3 — CGKA rotation driven by trie change

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
B (compute)MemberSoft—TraitImplSmall
C (recompute on rotation)MemberSoft—TraitImplSmall

What was found. BeeKEM exposes the cleanest CGKA substitution seam encountered in either spike:

// beekem/src/cgka.rs
pub async fn add<F, S>(
    &mut self,
    id: MemberId,
    pk: ShareKey,           // <-- direct key injection
    signer: &S,
) -> Result<Option<Signed<CgkaOperation>>, CgkaError>;

pub async fn update<F, S, R>(
    &mut self,
    new_pk: ShareKey,
    new_sk: ShareSecretKey,
    signer: &S,
    csprng: &mut R,
) -> Result<(PcsKey, Signed<CgkaOperation>), CgkaError>;

// keyhive_core/src/keyhive.rs
pub async fn force_pcs_update(&self, doc: Arc<Mutex<Document>>)
    -> Result<Signed<CgkaOperation>, EncryptError>;

The key flows in as a direct argument — no generic-trait threading like p2panda's IdentityRegistry<ID, Y> + PreKeyRegistry pattern. The spike's resolver maps MemberId → ShareKey and feeds it directly.

Asymptotic advantage. BeeKEM's tree structure gives O(log n) per rotation; p2panda's DCGKA is O(n). For ODS-scale orgs:

Org sizeKeyhive (log₂ n)p2panda (n)
100~7 ops~100 ops
1 000~10 ops~1 000 ops
10 000~14 ops~10 000 ops

Compounds significantly under frequent revocation churn.

No integration finding equivalent to p2panda's Dcgka::remove-vs-update issue. Keyhive's API distinguishes them cleanly: Cgka::remove(id, signer) for removal; Cgka::update(new_pk, ...) for refresh.

Gate 4 — Organisation-as-pseudo-group principal

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
A (delegation)OrgSoft—TraitImplSmall
C (recompute on rotation)OrgSoft—TraitImplMedium phase3

Strongest single Keyhive-vs-p2panda differentiator in the Phase 1.d data. Gate 4 was Hard for both p2panda flows (requiring a fork to access GroupMember::Group at the auth layer). For Keyhive both flows are Soft with no fork.

What was found. Agent::Group(GroupId, Arc<Mutex<Group>>) is a first-class variant of Agent. Keyhive::add_member takes Agent<F, S, T, L> and accepts ANY variant with an identical call signature. Adding the org as a member of a doc is one call:

keyhive.add_member(
    Agent::Group(org_group_id, org_group_arc),
    &doc_membered,
    Access::Edit,
    &[],
).await?;

Document::transitive_members() -> HashMap<Identifier, (Agent, Access)> auto-resolves nested groups, so reading the doc's effective membership walks the org group's individual members transparently.

Flow C (rotation cascade) requires the spike to maintain a reverse-index org_group_id → Vec<DocumentId> so that when an individual is removed from the org, every doc the org has access to can be CGKA-rotated. This index lives in the spike's gate-4 module and is populated whenever grant_org is called. Effort: Phase 3 Medium (reflects the index-maintenance code paths).

Documented constraint. Same as p2panda — managerial actions must stay with individuals; the org pseudo-group is a Read/Edit-level grant. Admin status is on-chain (Phase 1.a domain), not in the local-first ACL.

Evidence: spike-keyhive/src/evidence/s4.md.

Gate 5 — Peer-to-peer connection policy

Sub-flowPrincipalSeverityFailing subcrateFix pathEffort
E1 (accept/reject)MemberSoft—TraitImplSmall
E2 (accept/reject)OrgSoft—TraitImplSmall
F1 (terminate on revocation)MemberSoft—TraitImplSmall
F2 (terminate on org rotation)OrgSoft—TraitImplSmall

What was found. Keyhive has no published transport (Beelay is not yet released at the pin). The spike implements an in-process PolicyManager session model and bridges it to Keyhive via MembershipListener::on_revocation:

// spike-keyhive/src/s5_p2p_policy.rs (sketch)
pub struct PolicyManager<R: MemberKeyResolver> {
    resolver: Arc<R>,
    adapter: IdAdapter,
    sessions: Mutex<HashMap<SessionId, SessionRecord>>,
}
impl<R> PolicyManager<R> {
    pub fn authorise_session(&self, peer_vk: &VerifyingKey, doc: DocumentId)
        -> Result<SessionId, PolicyError>;
    pub fn recheck_open_sessions(&self) -> usize;
}

The MembershipListener wired into Keyhive calls policy_mgr.recheck_open_sessions() from on_revocation. This is push-based: a revocation propagating through receive_static_event immediately triggers session rechecks. p2panda's spike used a pull-based periodic recheck because no equivalent push hook was available at the sync layer.

Reverse-lookup gap (shared with p2panda). adapter.member_id_for(peer_vk) only finds entries cached by a prior resolve call. A cold authorisation request from a peer the spike has never seen has no entry. Phase 3 needs a formal MemberKeyResolver::find_member_by_device(VerifyingKey) → Option<MemberId> method — this is a foundation-layer concern (spike-common), not a Keyhive-specific issue.

5. Cross-cutting observations

Reverse-lookup gap (foundation-layer)

Both gate 4 (org rotation cascade) and gate 5 (connection policy) need a reverse lookup — given a device VerifyingKey, return the owning MemberId. The MemberKeyResolver trait in spike-common has no such method. Workarounds exist (the spike's IdAdapter caches the mapping on first lookup; PolicyManager accepts unknown peers as PolicyError::UnknownPeer), but they're incomplete.

This is the same gap recorded by spike-p2panda. Phase 3 should add the method to the resolver trait. This is foundation-layer work, not library-specific.

Reverse-index for org-as-pseudo-group

Gate 4 Flow C needs to know which documents have the org as a member, so that a rotation in the org cascades correctly. Keyhive does not expose this query — it's the inverse of the delegation graph. The spike maintains the index in its own wrapper state, populated on every grant_org call. Phase 3 must either rely on the wrapper owning this index, or add a Keyhive-side API for the query.

Async signer / FutureForm generic threading

Keyhive's generics are dense: Keyhive<F, S, T, P, C, L, R> has seven type parameters. Most have defaults (ContentRef = [u8; 32], L = NoListener, etc.), so instantiation is manageable, but any spike code that holds a typed reference to a Keyhive<...> has to commit to all seven. Phase 3 should consider a type alias (type OdsKeyhive = Keyhive<FutureFormTokio, OdsSigner, ...>) per scope to keep call sites readable.

Alpha-stability risk

Keyhive is 0.0.0-alpha.3. Three recent commits in the last 10 (1116eaf transitive delegation, f162854 serialization methods, 7914692 Send-ability fixes) show active feature development. The public Rust API surface may break between alphas. Re-run the L1 battery before any production pin.

8. Recommendations for Phase 3

  1. Adopt the IdAdapter pattern verbatim. The ~30-line spike-keyhive/src/adapter.rs is production-ready in shape if not in robustness. Phase 3 should harden it (handle resolver races, cache eviction policy) but the design is sound.
  2. Add reverse lookup to MemberKeyResolver. Foundation-layer change in spike-common: fn find_member_by_device(&self, vk: &VerifyingKey) -> Option<MemberId>. Eliminates the spike's closure-injection workaround.
  3. Build the org→docs reverse index inside the wrapper. For gate 4 Flow C cascade. Maintain inside the same wrapper that owns the Keyhive instance — keeps the index consistent with the delegation graph by construction.
  4. Use type aliases for the Keyhive generic stack. Pin the seven type parameters per application scope: type OdsKeyhive = Keyhive<TokioFutureForm, OdsSigner, [u8; 32], Vec<u8>, OdsCiphertextStore, PolicyListener, OsRng>;. Improves readability significantly.
  5. Re-pin and re-run L1 before production commit. Keyhive is alpha. Lock to a tagged release or vendor the SHA in a workspace sidecar that runs CI against the pin.
  6. Bridge MembershipListener to the trie's revocation propagation. The listener fires on local-only delegation/revocation events. Phase 3 must additionally route on-chain trie revocations through Keyhive::receive_static_event(StaticEvent::Revoked(...)) so that the local listener fires for both sources symmetrically.
  7. Track BeeKEM's O(log n) advantage as a key Phase 3 design lever. For ODS use cases with frequent revocation churn, the asymptotic advantage compounds. If a phase-3 scaling decision becomes contentious, this is the citation.
